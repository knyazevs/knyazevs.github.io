# Эволюция системы: Strangler Fig, parallel run, decommissioning

Архитектура — функция текущих требований, и требования меняются. Самые сложные
архитектурные задачи — не «как построить с нуля», а «как поменять работающую
систему, не сломав её». Здесь — три главных паттерна постепенной эволюции.

## Strangler Fig — постепенное замещение

### Метафора и идея

Strangler Fig (фикус-душитель) — растение, которое обвивает дерево-хозяина,
постепенно растёт вокруг него и в итоге полностью замещает его, оставляя пустую
форму внутри. Фикус не убивает дерево сразу, а замещает по чуть-чуть.

Архитектурный паттерн с тем же названием (Martin Fowler, 2004): когда нужно
заменить старую систему на новую, не делать big-bang переключение, а **постепенно
замещать функциональность**, пока старая система не сведётся к нулю.

### Как это выглядит

Шаги:

1. **Перед старой системой ставится прокси/фасад** (балансер, gateway,
   reverse-proxy). Все запросы идут через него.
2. **Прокси по умолчанию роутит на старую систему.** Ничего не сломалось.
3. **Новая функциональность реализуется в новой системе.** Прокси роутит
   соответствующие запросы туда. Старая система этих запросов не получает.
4. **Постепенно функциональность переносится из старой системы в новую.**
   Прокси перерутится по мере готовности.
5. **Когда вся функциональность переехала** — старая система выключается, прокси
   убирается.

В каждый момент времени система работает. Никаких многомесячных «параллельных
веток разработки», никаких big-bang переключений.

### На практике

Типичная история — миграция с legacy-монолита на современную архитектуру
(новый язык, новый стек, новая БД).

Шаги:

1. **API Gateway между клиентами и монолитом.** Несколько недель — балансировка,
   проверка, что ничего не сломалось.
2. **Выделить наименее связанный модуль** (например, поиск товаров). Реализовать
   его отдельным сервисом на новом стеке. Gateway роутит `/api/search/*` в новый
   сервис.
3. **Если новый сервис работает стабильно** — следующий модуль. Если что-то идёт
   не так — gateway возвращает запросы в монолит, инцидент локальный.
4. **Через год-два** монолит — это огрызок с парой эндпоинтов, всё остальное
   переехало. Решаем — выпилить остатки или оставить.

Известные примеры:
- **Shopify** в 2020-2022 переезжали с большого Ruby-монолита на microservices,
  используя Strangler Fig
- **Amazon** в 2000-х переходили с C++ монолита на сервисы — постепенно, по
  модулям

### Когда модель усложняется

**Shared state между старой и новой системами.** Если новая система пишет в свою
БД, а часть запросов всё ещё ходит в старую (тоже пишущую) — нужна синхронизация
данных в обе стороны.

Решения:
- **Dual write** (оба сервиса пишут в обе БД) — рискованно, легко получить
  расхождение
- **CDC** (изменения из одной БД стримятся в другую через Debezium или подобное)
  — асинхронно, eventual consistency
- **Sharding по ключу** — какой-то набор пользователей читает/пишет в новую
  систему, остальные в старую; постепенный rollout по %

Shared state — самая сложная часть Strangler Fig. Если нет shared state, миграция
тривиальная. Если есть — это многомесячный проект сам по себе.

## Parallel run — снижение риска большой миграции

### Идея

Когда нужно поменять критическую логику (например, расчёт цены, начисление
бонусов, ML-модель), нельзя просто запустить новую версию и надеяться, что она
правильная.

**Parallel run** — новая логика работает **параллельно** со старой, но её ответ
**не показывается пользователю**. Только старый ответ идёт в продакшн. Параллельно
оба ответа сравниваются и логируются.

После периода сравнения, когда новая логика стабильно даёт одинаковые с старой
результаты (или различия признаются ожидаемыми и правильными), новая логика
становится основной, старая отключается.

### Как это выглядит

```
              ┌─────────────┐
   запрос →  ─┤ контроллер  ├─→ старая логика → ответ → клиенту
              │             │ ↘
              │             │   новая логика → ответ → отбрасывается
              │             │                       ↓
              │             │                  лог сравнения
              └─────────────┘
```

В логе сравнения: запрос, ответ старой, ответ новой, идентичны ли, разница (если
есть). Аналитики смотрят на эти данные, разбираются с расхождениями, либо
исправляют новую логику, либо уточняют тестовые случаи.

### На практике

Где особенно полезен parallel run:

- **Финансовые расчёты.** Замена логики начисления комиссий, налогов, скидок.
  Нельзя ошибиться, пользовательские деньги.
- **Pricing engines.** Новая модель ценообразования.
- **ML-модели в проде.** Новая модель ранжирования / рекомендаций — сначала
  параллельно, потом переключение.
- **Миграция между БД.** Старая и новая БД пишутся одновременно, читается из
  старой; после стабилизации — читается из новой.

Период parallel run — недели или месяцы. Достаточно, чтобы покрыть все edge cases
(включая редкие — сезонные пики, конкретные категории товаров).

### Когда модель усложняется

**Стоимость parallel run** значительная — двойная нагрузка на инфраструктуру,
двойное логирование, время инженеров на разбор расхождений. Имеет смысл только
для критичной логики, где цена ошибки превышает цену parallel run.

**Side effects.** Если новая и старая логика обе делают side effects (отправляют
письма, делают платежи), parallel run опасен — пользователь получит два письма
или два платежа. Решение: parallel run только для **чистой** логики (расчёт без
side effects), либо хитрая схема, где side effects делает только одна из логик.

**Различия — это нормально, иногда правильно.** Цель миграции — обычно не
бит-в-бит совпадение, а **корректность**. Новая логика может давать другие
результаты на edge case'ах, и это правильно — старая была багнутая. Аналитика
должна различать «новая хуже» от «новая правильнее».

## Decommissioning — как выпиливать legacy

### Идея

Когда систему пора выключить (миграция завершена, продукт устарел, технология
deprecated), правильное decommissioning — это **процесс**, а не одна команда
`shutdown`.

Цель: выключить систему так, чтобы:
- Никто из пользователей не пострадал внезапно
- Данные не потерялись
- Зависимые системы не сломались
- Информация осталась доступна для аудита/восстановления

### Шаги

1. **Inventory.** Что вообще представляет собой эта система? Какие endpoint'ы,
   какие данные, какие зависимости (входящие и исходящие)?
2. **Identification of users.** Кто пользуется? Часто оказывается, что система
   «никому не нужна», но потом выясняется, что отдел финансов раз в квартал
   запускает отчёт.
3. **Communication.** Объявление deprecation с датой выключения. Минимум 3-6
   месяцев предупреждения для критичных систем.
4. **Migration support.** Помочь пользователям перейти на новую систему. Документация,
   one-on-one сессии для крупных пользователей.
5. **Read-only mode.** Перевести систему в режим «только чтение» — никто не пишет
   новые данные, старые ещё доступны. Видно, кто всё ещё активно использует.
6. **Data archival.** Снять полный snapshot данных, сохранить в холодное хранилище
   (S3 Glacier, etc.). Документировать формат и схему.
7. **Shutdown.** Выключение сервиса. Обычно через redirect на новую систему или
   sunset-страницу.
8. **Resource cleanup.** Удалить инфраструктуру, IAM-роли, домены, секреты,
   мониторинг. Иначе через год они станут «загадочными ресурсами без владельца».

### На практике

Типичная ошибка — пропустить шаги 5-6. Команда «выключила» сервис, а потом через
полгода выясняется, что нужны данные за прошлый год для compliance-аудита.

Лучший подход — **продумывать decommissioning как часть миграции**, не как
отдельную задачу в конце. План миграции должен включать процесс выключения с
датами и ответственными.

### Когда модель усложняется

**Long tail of legacy users.** Иногда выключение «откладывается на следующий
квартал» снова и снова, потому что есть один-два важных клиента, которые не
мигрировали. Это превращает decommissioning в бесконечный процесс.

Решения:
- **Жёсткий cutoff date** — published заранее, не двигается. Клиенты знают и
  планируют.
- **Платная поддержка legacy** — после cutoff клиенты могут продолжать использовать
  за отдельную плату. Это либо мотивирует мигрировать, либо окупает поддержку.
- **Ручной snapshot для last users** — если 1-2 важных клиента — может быть
  проще сделать им dump данных и помочь интегрироваться с новой системой
  отдельно, чем держать всю инфраструктуру.

### Где деньги

Legacy-системы стоят деньги даже когда «никто их не использует»:

- **Hosting и инфраструктура** — БД, серверы, мониторинг
- **Security** — каждая система должна патчиться, сертифицироваться
- **Dependency** — если она держит общую БД или интеграцию, это ограничивает
  изменения вокруг
- **Cognitive load** — новые инженеры спрашивают «что это», им объясняют,
  поддерживают документацию

Грубая оценка: каждая legacy-система обходится в $50-100K/год скрытых издержек
даже без активной разработки. Decommissioning — это **экономическая инвестиция**,
не очистка ради чистоты.

### Когда оставлять legacy

Бывают случаи, когда decommissioning **не оправдан**:

- **Compliance requires data retention** — если нужно хранить данные 7 лет для
  аудита, проще оставить read-only систему, чем мигрировать данные.
- **Очень редкое, но критичное использование** — например, пенсионные расчёты,
  которые делаются раз в год.
- **Stable, low-maintenance** — система давно стабильна, никаких изменений,
  cost замены превышает cost поддержки.

В этих случаях — оставить, но **минимизировать поверхность**: read-only,
иссушенный feature set, минимальная инфраструктура.
