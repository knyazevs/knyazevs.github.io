# Контракты: REST vs GraphQL vs gRPC, версионирование, обратная совместимость

API — это контракт между сервисами или между сервисом и клиентами. Выбор
протокола, способа версионирования и стратегии эволюции определяет, насколько
больно тебе будет менять систему через год. Здесь — про trade-off'ы трёх главных
протоколов и про правила эволюции контракта без слома клиентов.

## REST vs GraphQL vs gRPC — когда что выбирать

### REST

Самый распространённый. Ресурсы (URL'ы), операции (HTTP-методы), JSON в теле.

**Сильные стороны:**

- **Универсальность.** Работает в любом языке, любом браузере, любой инструмент
  поддерживает.
- **Кешируемость на уровне HTTP.** GET-запросы кешируются прокси, CDN, браузером
  без специальных усилий.
- **Простота для внешних клиентов.** Сторонний разработчик за час разберётся, как
  делать запросы curl'ом.
- **Хорошо документируется через OpenAPI** (бывший Swagger). Контракт описывается
  как документ, генерируются клиенты.

**Слабые стороны:**

- **Over-fetching и under-fetching.** Endpoint возвращает либо слишком много (мобильный
  клиент тащит лишние поля), либо слишком мало (приходится делать N дополнительных
  запросов).
- **Множество round-trip'ов.** Чтобы получить заказ + товары + адрес — три запроса,
  каждый со своим latency.
- **Нет строгой типизации в стандарте.** OpenAPI помогает, но это слой поверх,
  не часть протокола.

**Когда выбирать:**
- Публичные API для сторонних разработчиков
- Простые backend → frontend сценарии
- Когда CDN-кеширование критично
- По умолчанию для большинства задач

### GraphQL

Один endpoint, клиент описывает в запросе, какие поля ему нужны, сервер собирает
ответ.

**Сильные стороны:**

- **Клиент берёт ровно то, что нужно.** Решает over-fetching проблему. Особенно
  ценно для мобильных клиентов на медленной сети.
- **Один запрос — несколько ресурсов.** Заказ + товары + адрес одним запросом,
  один round-trip.
- **Строгая типизация контракта.** Schema — обязательная часть, генерируются
  типизированные клиенты.
- **Эволюция через deprecation** полей — клиенты, не использующие старое поле,
  не пострадают.

**Слабые стороны:**

- **Сложно кешировать.** Каждый запрос уникальный, HTTP-кеширование не работает
  как с REST. Нужны GraphQL-специфичные кеши (Apollo, Relay).
- **N+1 проблема на сервере.** Resolver для каждого поля может ходить в БД —
  легко получить 100 запросов на один GraphQL-запрос. Решается DataLoader'ами,
  но требует осознанного проектирования.
- **Сложность авторизации.** Каждое поле может требовать своих прав; сложнее, чем
  «этот endpoint требует роль X».
- **Барьер входа для внешних клиентов.** Сложнее explore'ить, чем REST.

**Когда выбирать:**
- Backend-for-Frontend для разнотипных клиентов (мобайл + веб + админка с разными
  потребностями)
- Сложные UI с большим количеством взаимосвязанных данных
- Внутренние API между frontend-командой и backend-командой
- НЕ для публичных API сторонним

### gRPC

Бинарный протокол поверх HTTP/2. Контракт описан в .proto файлах. Генерация
клиентов и серверов из proto.

**Сильные стороны:**

- **Бинарный, компактный.** В разы меньше JSON по объёму.
- **HTTP/2 multiplexing.** Много запросов в одном TCP-соединении, без head-of-line
  blocking.
- **Streaming.** Поддерживается server-streaming, client-streaming, bidirectional
  streaming — для long-polling, чатов, real-time.
- **Строгая типизация и кодогенерация** для большинства языков. Контракт
  обновился — клиенты пересобрались.
- **Performance.** В разы быстрее REST на больших объёмах.

**Слабые стороны:**

- **Браузеры не работают напрямую.** Нужен gRPC-Web (proxy через Envoy) или
  отдельный REST-фасад для веба.
- **Сложность отладки.** Curl'ом не потыкаешь, нужны специальные инструменты
  (grpcurl, BloomRPC).
- **Tooling beyond standard стеков** ограничен.

**Когда выбирать:**
- Внутренние сервис-to-сервис коммуникации (микросервисы между собой)
- Высоконагруженные системы, где экономия байтов и round-trip'ов критична
- Streaming-сценарии (real-time данные, long-running операции)
- НЕ для публичных API и НЕ для прямой работы с браузером

### На практике

Гибрид — нормальная стратегия:

- **Внутри backend** — gRPC между сервисами (производительность, типизация)
- **Backend → Web/Mobile** — GraphQL или REST в зависимости от UI complexity
- **Public API** — REST с OpenAPI (универсальность)

Так делает Netflix, Slack, Shopify и многие другие. Один протокол на всё —
вынужденная унификация ради простоты, не оптимум.

## Versioning — стратегии эволюции

### Зачем версионировать

Контракт API меняется. Добавляются поля, удаляются эндпоинты, меняется семантика.
Если делать изменения «вживую», ломается каждый клиент в момент твоего деплоя.

Версионирование позволяет **существовать одновременно нескольким версиям**, давая
клиентам время мигрировать.

### Способы версионирования

**1. URL versioning** — `/api/v1/orders`, `/api/v2/orders`.

- Плюсы: явно, понятно где какая версия, легко роутить балансером.
- Минусы: при росте версий — много дублирования URL и handler'ов; клиент должен
  знать «свою версию» в URL.

**2. Header versioning** — `Accept: application/vnd.company.v2+json`.

- Плюсы: URL чистый, версия — деталь протокола.
- Минусы: труднее тестировать (нужно явно ставить header), harder для CDN-кеша.

**3. Query parameter** — `/api/orders?version=2`.

- Плюсы: легко тестировать через браузер, возможен fallback на дефолтную версию.
- Минусы: засоряет URL, проблемы с кешированием.

**4. Без версий — только аддитивные изменения.**

- Никогда не делать breaking changes; только добавлять поля и эндпоинты.
- Удаления — через долгий deprecation период.
- Плюсы: нет накладных расходов на версии.
- Минусы: ограничивает эволюцию, накапливаются legacy-поля.

### На практике

Большинство публичных REST API используют URL versioning (Stripe v1, GitHub v3, etc.).
Большинство внутренних API не версионируются явно, а полагаются на принцип
аддитивности (никогда не ломать).

**Внутренние сервисы** в микросервисной архитектуре обычно используют **версионирование
сообщений** (event schemas), а не версии endpoint'ов. Например, Avro Schema
Registry для Kafka — каждое событие имеет версию схемы, новая схема должна быть
backwards-compatible со старой.

### Когда модель усложняется

**Несколько версий параллельно** — сложнее, чем кажется. Если v1 и v2 пишут в
одну БД, надо аккуратно обрабатывать поля, которые есть в одной версии, но нет в
другой. Часто проще иметь два разных endpoint, чем «один с условиями».

**Удаление старой версии** — отдельный organizational challenge. Нужно:
1. Объявить deprecation с датой
2. Мониторить, кто ещё использует
3. Связаться с пользователями, помочь мигрировать
4. После всех — удалить

Stripe знаменит тем, что **никогда не удаляет старые версии API**. У них до сих
пор работают эндпоинты от 2011 года. Это сознательная инвестиция в обратную
совместимость как продуктовую фичу — клиент может не обновляться годами и не
сломается.

## Эволюция без слома — правила

### Backwards-compatible изменения (можно делать всегда)

- **Добавить опциональное поле в response.** Клиенты, не знающие про поле,
  игнорируют его.
- **Добавить опциональное поле в request.** Старые клиенты не присылают —
  сервер использует default.
- **Добавить новый endpoint.** Никого не задевает.
- **Расширить enum новыми значениями.** Клиенты должны знать, что enum может
  расширяться (handler-default «unknown value»).

### Breaking changes (нельзя делать молча)

- **Удалить поле из response.** Клиент, читавший поле, упадёт.
- **Сделать опциональное поле обязательным в request.** Старые клиенты, не
  присылающие поле, получат ошибку.
- **Поменять тип поля.** `id: int → id: string` — полная катастрофа.
- **Поменять семантику.** То же поле, тот же тип, но другое значение —
  невидимое breakage. Худший вид.
- **Удалить значение из enum.** Клиент, отправивший старое значение, получит
  ошибку.

### Стратегия эволюции

Для breaking changes:

1. **Добавить новое.** Поле/endpoint/значение enum появляется в новой форме.
2. **Старое помечается deprecated.** В документации, в response (где можно),
   в логах сервера (warning при использовании).
3. **Метрика использования.** Кто и насколько активно использует deprecated.
4. **Период миграции.** Месяцы или годы, в зависимости от важности клиентов.
5. **Удаление.** Только после того, как метрика упала до нуля.

### На практике

Reaction на «у нас релиз через две недели, надо удалить старое поле» — должна быть
**отказ**. Удаление поля без предварительного deprecation — это контракт-нарушение,
независимо от срока.

Приёмы:

- **Schema-first design** — контракт пишется первым, в OpenAPI/proto. Изменения
  обсуждаются как изменения схемы, не как изменения кода. Помогает увидеть
  breaking changes явно.
- **Contract testing** (Pact, Spring Cloud Contract) — клиент описывает свой
  expected ответ, сервер проверяет, что не ломает его. Catch breaking changes в CI.
- **Schema Registry** для событий (Confluent Schema Registry для Avro) —
  enforced backwards-compatibility на уровне инфраструктуры, нельзя залить
  несовместимую схему.

### Когда модель усложняется

**Internal vs external API** — разные правила.

Внутренние API между сервисами одной компании — можно координированно поменять
обе стороны. Deprecation короче, иногда даже big-bang миграция.

External API для сторонних клиентов — никаких big-bang, только аддитивно. Если
нужен breaking change — это новая major-версия (`/v2/`) с долгим co-existence с
v1.

### Где деньги

Backwards-compatibility — это **долгосрочная инвестиция в трастовые отношения с
клиентами**. Каждый раз, когда ты ломаешь чей-то клиент — ты теряешь его
доверие, и он начинает закладывать буфер на твои изменения (тестирование,
отдельные релизы под твои обновления).

Stripe, Slack, GitHub — компании с лучшей репутацией developer experience —
тратят значительные ресурсы на сохранение совместимости. Это маркетинговое
преимущество, не overhead.

Книги для углубления:
- *Building Evolutionary Architectures* (Neal Ford et al.) — про эволюцию
  систем во времени
- *Designing Web APIs* (Brenda Jin et al.) — практика дизайна публичных API
- Stripe API documentation — образец documentation и stability
