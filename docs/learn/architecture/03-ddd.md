# DDD на практике: bounded contexts, агрегаты, ubiquitous language

Domain-Driven Design (DDD, проектирование от предметной области) — методология,
которую часто понимают неправильно. Чаще всего её сводят к набору паттернов
(Entity, Value Object, Repository), упуская главное. Главное в DDD — **умение
находить правильные границы в бизнес-домене и моделировать систему вокруг них**.

## Bounded contexts — главный концепт DDD

### Что это

**Bounded context** (ограниченный контекст) — кусок системы, в котором определённые
бизнес-понятия имеют **конкретное значение**. За пределами контекста те же слова
могут значить другое.

Классический пример: понятие «Заказ» в e-commerce.

В контексте `Checkout` (оформление):
- Заказ = состояние корзины + адрес + способ оплаты
- Атрибуты: список товаров с количествами и ценами, скидки, итоговая сумма
- Операции: добавить товар, применить промокод, разместить

В контексте `Shipping` (доставка):
- Заказ = груз с адресом
- Атрибуты: вес, габариты, адрес, выбранный курьер, статус доставки
- Операции: запланировать, отгрузить, отметить доставленным

В контексте `Billing`:
- Заказ = факт начисления
- Атрибуты: сумма, статус оплаты, метод
- Операции: списать, рефанд, начисление

Это **три разные модели** одной концепции. Каждая модель оптимальна для своего
контекста. Никакая «общая модель Заказа на всю систему» не подходит — она будет
содержать поля для всех контекстов одновременно (anemic domain model) и не будет
ни одному контексту по-настоящему удобна.

### Почему это важно

Каждый bounded context — это **кандидат в отдельный сервис** (если декомпозиция
оправдана) или **отдельный модуль** (в monolith'е). Границы сервисов = границы
контекстов.

Без bounded contexts ты получаешь:
- Огромную «универсальную» модель Заказа с 50 полями (потому что все контексты
  внесли свои)
- Невозможность поменять что-то в одном контексте без риска сломать другой
- Бесконечные споры команд «правильное» имя для атрибута

С bounded contexts:
- Каждый контекст имеет свою модель, оптимальную для своих задач
- Изменения в одном контексте не задевают другие
- Имена и значения — однозначные внутри контекста

### На практике

Как находить bounded contexts:

- **Смотри на язык бизнеса.** Если бизнес использует разные слова в разных
  ситуациях для близких концепций — это сигнал разных контекстов.
- **Смотри на команды.** Если за разные части одного домена отвечают разные
  команды — обычно это разные контексты.
- **Смотри на жизненный цикл.** Заказ в `Checkout` живёт минуты-часы, в
  `Shipping` — дни, в `Billing` — годы (для аудита). Разные циклы → разные
  контексты.
- **Смотри на правила.** В одном контексте «Заказ можно отменить за 30 минут»,
  в другом «Заказ нельзя отменить после отгрузки». Если правила разные — модель
  тоже разная.

### Когда модель усложняется

**Anti-corruption layer (ACL)** — паттерн на границе двух контекстов. Когда два
контекста взаимодействуют, между ними ставится тонкий адаптер, который переводит
понятия одной модели в понятия другой. Не пускает «грязь» одной модели в другую.

Пример: `Checkout` создаёт заказ, отправляет событие `OrderPlaced` в `Shipping`.
`Shipping` не получает «всю модель заказа из Checkout» — он получает только то,
что нужно ему: id, адрес, список товаров с весами. ACL внутри `Shipping`
конструирует свою модель из этого минимума.

Это не «дублирование данных», это **разделение ответственности**. `Shipping`
самостоятелен в своей модели и не зависит от изменений в `Checkout`.

## Aggregates, Entities, Value Objects на пальцах

### Зачем эти понятия

DDD предлагает три тактических паттерна для моделирования внутри bounded context:
**Entity**, **Value Object**, **Aggregate**. Их часто учат как «объекты с id»,
«объекты без id», «корневые объекты» — это правда, но это не главное.

Главное — они помогают **очертить транзакционные границы и инварианты**.

### Entity

**Entity** (сущность) — объект, который имеет идентичность во времени. Меняется,
но остаётся «тем же». Имеет id.

Примеры: Заказ, Пользователь, Товар. Заказ может менять состояние (создан →
оплачен → отгружен), но это всё тот же Заказ #12345.

Entity сравниваются по id, не по содержимому. Два Заказа с одинаковыми полями
— это **разные** Заказы, если у них разные id.

### Value Object

**Value Object** (значимый объект) — объект, который определяется только своим
значением. Не имеет идентичности.

Примеры:
- Деньги (`Money(amount=100, currency=USD)`)
- Адрес (`Address(street, city, zip)`)
- Период (`Period(from, to)`)
- Координата (`GeoPoint(lat, lng)`)

Value Object'ы сравниваются по значениям. Два адреса с одинаковыми полями — это
**один и тот же** адрес. Value Object'ы обычно immutable — изменение даёт новый
объект, не меняет старый.

Не путай с DTO: Value Object имеет **поведение** (Money умеет складываться,
учитывать валюту, сравниваться), DTO — просто структура для переноса данных.

### Aggregate

**Aggregate** (агрегат) — группа связанных Entity и Value Object, которые меняются
**атомарно** и подчиняются общим **инвариантам** (правилам, которые всегда должны
выполняться).

У агрегата есть **корень (Aggregate Root)** — единственная Entity, через которую
происходит доступ извне. Внутренние объекты агрегата нельзя менять напрямую —
только через корень.

Пример: Order Aggregate.
- Корень: Order (Entity с id)
- Внутренние Entity: OrderItem (с id внутри Order)
- Value Objects: ShippingAddress, Money (для total)
- Инвариант: «Сумма всех OrderItem = total Order»; «Order не может быть пустым»;
  «После статуса PAID нельзя добавлять items»

Извне: код видит только Order. Если хочешь добавить item — `order.addItem(...)`,
а не напрямую в коллекцию items. Внутри метода Order проверяет инварианты и не
даст добавить item к оплаченному заказу.

### Транзакционная граница

Главное правило: **одна транзакция = один агрегат**.

Если транзакции нужно менять два агрегата — это сигнал, что либо они должны быть
одним агрегатом (если сильно связаны), либо взаимодействовать через события
(eventual consistency).

Пример: «оформить заказ» — это операция над Order агрегатом и Cart агрегатом
(или Inventory агрегатом). Их нельзя менять в одной транзакции; нужна
последовательность с компенсацией при ошибке (saga).

### На практике

Типичные ошибки:

- **God Aggregate.** Один огромный агрегат на всё (User содержит Orders, Carts,
  Reviews). Тяжело загрузить, концерн один (User), но логика рассыпана. Решение:
  выделить отдельные агрегаты по транзакционным границам.
- **Слишком мелкие агрегаты.** Каждая Entity — свой агрегат, инварианты не
  проверяются. Решение: посмотреть на правила — что должно быть атомарно меняться
  вместе?
- **Прямая мутация внутренностей.** `order.items.add(item)` извне, без проверок
  инвариантов в Order. Решение: incapsulate коллекции, делать `order.addItem(item)`
  с проверками внутри.

### Когда модель усложняется

Иногда инварианты охватывают **несколько агрегатов** — например, «общая стоимость
всех заказов клиента не должна превышать кредитный лимит». Это не помещается в
один агрегат естественно.

Решения:
- **Domain service.** Отдельный класс, который принимает несколько агрегатов и
  координирует операцию. Не имеет состояния, только логика.
- **Eventual consistency.** Создать заказ, асинхронно проверить лимит, при
  превышении — компенсация (отмена заказа).
- **Snapshot.** Хранить «общий лимит» как поле в Customer агрегате, обновлять
  при создании/отмене заказа.

Какое решение правильное — зависит от строгости инварианта. Деньги — обычно
строго, нужен domain service или snapshot. UI-ограничение типа «не более 10
заказов в день» — eventual consistency хватит.

## Ubiquitous language — общий язык команды и бизнеса

### Что это

**Ubiquitous language** (вездесущий язык) — общий словарь, которым пользуются
**и инженеры, и бизнес** при разговоре о домене.

Не «то, что говорят бизнес» (язык бизнеса часто неточный) и не «технический жаргон
инженеров». А **согласованный словарь**, выработанный совместно.

Пример. Бизнес говорит «оформить покупку». Инженеры пишут код с термином
`createOrder()`. Дизайнеры рисуют кнопку «Оформить заказ». Поддержка в чате
говорит «оплачено? отгружено?». Все эти термины должны быть **в одном словаре**:
- Оформить заказ = checkout
- Покупка → Заказ (Order)
- Состояния: создан, оплачен, отгружен, доставлен

И всё это должно быть в коде:
```
fun checkout(cart: Cart): Order  // не createOrder, не buyCart, не purchaseItems
class Order(state: OrderState)  // OrderState.CREATED, PAID, SHIPPED, DELIVERED
```

### Зачем

Главная боль архитектуры — **расхождение между моделью бизнеса и моделью кода**.
Когда бизнес говорит одно, код говорит другое, и постоянная переводчица между
ними — единственный носитель знания, на которого все полагаются.

Ubiquitous language убирает эту проблему: **код — это документ домена**. Если
бизнес читает код (или хотя бы имена классов и методов), он узнаёт свой домен.

### На практике

Как вырабатывать ubiquitous language:

- **Event Storming workshop.** Соберите бизнес и инженеров в одной комнате (или
  Miro), накидайте стикеры с событиями домена в порядке времени. Из дискуссии
  выкристаллизуется словарь.
- **Glossary в репозитории.** Файл `GLOSSARY.md` или Confluence-страница, где
  описаны все термины с определениями. Поддерживается как код.
- **Code review на терминологию.** Если в код попадает `processOrder()` вместо
  установленного `fulfillOrder()` — реджект.
- **Refactoring при смене языка.** Если бизнес начал говорить «Опера» вместо
  «Сделка» — переименовать в коде. Storage migration — отдельная задача, но имена
  должны соответствовать.

### Когда модель усложняется

Внутри одного домена могут быть **несколько контекстов с разным языком**. «Заказ»
в `Checkout` и «Заказ» в `Shipping` — могут быть разными терминами для разных
команд. Это не проблема — это и есть смысл bounded context'ов.

Каждый bounded context имеет **свой** ubiquitous language. На границах контекстов
(в anti-corruption layer) происходит перевод.

Книги для углубления:
- *Domain-Driven Design* (Eric Evans, 2003) — каноническая, толстая, читать
  выборочно
- *Implementing Domain-Driven Design* (Vaughn Vernon, 2013) — более практическая,
  с примерами кода
- *Domain-Driven Design Distilled* (Vaughn Vernon, 2016) — короткая, для
  знакомства за день
- Event Storming — материалы Alberto Brandolini, methodology для discovery
  bounded contexts
