# Декомпозиция сервисов: границы, владение данными, Conway's Law

Если решено разделять систему на сервисы, главный вопрос — **где резать**.
Ответ почти никогда не «по техническим слоям», и почти всегда — «по бизнес-границам
с учётом структуры команд». Здесь — про критерии границ, владение данными и
Conway's Law как закон, который ты не отменишь.

## Где резать — бизнес-границы, не технические слои

### Анти-паттерн «техническое разделение»

Самый частый плохой подход к декомпозиции:

- `frontend-service` — отдаёт HTML/JSON
- `backend-service` — бизнес-логика
- `data-service` — работа с БД

Это **разделение по техническим слоям** (presentation/business/data). Звучит
аккуратно, но создаёт distributed monolith: каждый запрос пользователя проходит
через все три сервиса, ни один не имеет смысла отдельно, любое изменение фичи
требует деплоя всех трёх.

Аналогично:
- `validation-service`, `transformation-service`, `notification-service` — слои
  ETL-пайплайна как отдельные сервисы
- `auth-service` для каждого нюанса аутентификации (логин, регистрация, сессии)

Все эти разделения создают сетевую связанность вместо пользы.

### Правильный подход — по бизнес-возможностям

Разделение по **бизнес-возможностям** (business capabilities) или **поддоменам**.

Признаки правильной границы:
- **Сервис делает что-то осмысленное для бизнеса.** «Управление заказами»,
  «Каталог товаров», «Биллинг», «Уведомления».
- **Сервис может работать в изоляции.** Если все остальные сервисы упали, этот
  должен иметь возможность отвечать хотя бы по своему функционалу.
- **Сервис меняется по своим причинам.** Изменение в логике биллинга не должно
  требовать изменения в каталоге товаров.
- **Команда (или подкоманда) может владеть им.** Один сервис — не больше одной
  команды. Иначе теряется владение.

Канонический пример из e-commerce:
- `Catalog` — каталог товаров, поиск, фильтры
- `Cart` — корзина пользователя
- `Checkout` — оформление заказа
- `Payments` — приём платежей
- `Inventory` — учёт остатков
- `Shipping` — доставка
- `Notifications` — уведомления (email, push)

Каждый имеет свой бизнес-смысл, свою БД, свою команду.

### На практике

Когда смотришь на систему и думаешь, как её разделить — не смотри на код. Смотри
на **бизнес-домен**:

- **Какие у нас процессы?** Каждый процесс — потенциальный сервис.
- **Какие сущности?** Каждая «корневая сущность» (заказ, пользователь, товар) —
  обычно центр своего сервиса.
- **Кто отвечает за что в организации?** Команда поддержки заказов, команда
  каталога, команда платежей — это уже карта декомпозиции.

Подход называется **Domain-Driven Design** (DDD), подробно в главе 03.

### Когда модель усложняется

Бизнес-границы бывают **размытыми**. Заказ — это часть `Cart`, `Checkout`,
`Payments`, `Shipping`? Какая команда «владеет» Заказом?

Решения:
- **Контекст вокруг сущности.** Один и тот же «Заказ» в `Checkout` — это
  оформляемый заказ с ценой и адресом. В `Shipping` — это пакет с весом и адресом.
  Это **разные модели** одной концепции, и каждый сервис владеет своей моделью.
  Понятие из DDD — **bounded context** (ограниченный контекст).
- **Один источник истины (system of record) для каждого факта.** Цена товара
  определяется в `Catalog`, не в `Cart`. Адрес доставки определяется на
  `Checkout`, не в `Profile`. Если разные сервисы хранят дублирующую информацию,
  один из них должен быть «правдой», остальные — кешами или копиями.
- **События как клей.** `Checkout` публикует событие `OrderPlaced`, его слушают
  `Inventory` (списать товар), `Payments` (взять деньги), `Shipping` (запланировать
  доставку), `Notifications` (написать пользователю). Каждый сервис реагирует
  по-своему, нет единого «оркестратора», который знает всё.

## Владение данными — почему shared DB убивает микросервисы

### Правило: одна БД на сервис

Каждый сервис **владеет своей БД**. Никто другой к ней напрямую не обращается.
Если другому сервису нужны данные — через API владельца, не через SQL.

Почему так строго:

- **Схема БД — это контракт.** Если несколько сервисов читают/пишут одну таблицу
  напрямую, ты не можешь поменять структуру таблицы без координации со всеми
  читателями. Schema migration становится организационным проектом.
- **Транзакции через границы — иллюзия.** Два сервиса, пишущие в одну БД, могут
  блокировать друг друга, deadlock'ить, видеть промежуточные состояния. Это
  возвращает все проблемы distributed transactions, плюс ничего не выигрывает.
- **Разные сервисы — разные паттерны нагрузки.** Один — write-heavy транзакции,
  другой — analytical queries. Разные индексы, разные настройки, разные потребности
  в репликации. Одна БД на двоих — всегда компромисс не в пользу обоих.

### На практике

Когда в существующей системе уже есть shared DB между сервисами, расщепление —
многомесячная работа. Обычные шаги:

1. **Identify ownership.** Для каждой таблицы определить, какой сервис «настоящий
   владелец». Остальные — клиенты.
2. **Wrapping API.** Владелец публикует API над своими таблицами. Остальные
   сервисы переключаются на API вместо прямых SQL-запросов.
3. **Physical separation.** После того как все сервисы перешли на API, таблицы
   физически переезжают в отдельную БД владельца.

Это многоэтапная миграция. Подробно — в главе 06 про Strangler Fig.

### Когда нужно читать чужие данные

Иногда сервису нужно **много** читать чужие данные — например, для составления
отчёта или аналитики. Вызывать API в цикле — плохо.

Решения:

- **Materialized view / read model.** Сервис подписывается на события владельца
  и строит свою локальную копию данных в нужной форме. Eventual consistency, но
  чтение из своей БД.
- **CDC (Change Data Capture).** Изменения в БД владельца стримятся в Kafka, другие
  сервисы могут собирать свои read-модели. Tools: Debezium, AWS DMS.
- **Аналитический warehouse.** Все сервисы стримят данные в общий warehouse
  (Snowflake, BigQuery), который не используется для оперативной работы. Аналитика
  делается там.

Главное: **ни в каком из этих вариантов чужой сервис не лезет напрямую в чужую БД**.
Связанность через события, не через схему.

## Conway's Law — структура команд диктует архитектуру

### Что это

Закон Конвея (1968):

> Любая организация, проектирующая систему, неизбежно создаст дизайн, чья
> структура повторяет структуру коммуникации в этой организации.

Перевод на нормальный язык: **структура твоих сервисов будет похожа на структуру
твоих команд**. Если у тебя три команды, у тебя будут три (логических) сервиса
или модуля. Если у тебя одна команда — будет один монолит, как бы ты ни старался
разделить.

### Inverse Conway Maneuver

Если ты хочешь определённую архитектуру (например, 5 микросервисов), сначала
организуй **5 команд**, владеющих этими сервисами. Иначе закон Конвея вернётся и
сожрёт твою красивую архитектуру: команды найдут способы сделать жёсткую
координацию через сетевые контракты.

Это и называется **inverse Conway maneuver** — менять структуру команд под
желаемую архитектуру, а не наоборот.

Пример: AWS внутри организован по принципу «two-pizza teams» (команда не больше
8-10 человек), и каждая команда owns свой сервис end-to-end. Это и формирует
архитектурный паттерн микросервисов в AWS.

### На практике

Перед декомпозицией сервисов **посмотри на команды**:

- **Если у тебя одна команда из 5 человек** — не дробь систему на 10 сервисов.
  Ты не выдержишь координацию. Останься в монолите.
- **Если у тебя несколько команд по 5-8 человек** — каждая команда может owns
  один-два сервиса. Декомпозиция следует структуре команд.
- **Если ты планируешь рост до многих команд** — decomposition должен быть
  заложен заранее, через modular monolith с жёсткими границами или через
  выделение самых независимых сервисов.

Худший вариант: декомпозировать систему на 20 сервисов, оставив 10 инженеров.
Каждое изменение требует трогать 5 сервисов, координация невозможна, никто не
владеет ничем end-to-end.

### Когда модель усложняется

Иногда архитектура **должна** не следовать командам — например, есть
cross-cutting concerns (auth, logging, deployment), которыми занимается
платформенная команда. Здесь:

- **Платформенные сервисы** (auth, payment, observability) — обычно общие, но с
  чёткими API
- **Платформенная команда** — отдельная, владеет инфраструктурой и shared
  библиотеками
- **Продуктовые команды** — owns свои сервисы, потребляют платформенные

Это всё ещё Conway: продуктовые сервисы соответствуют продуктовым командам,
платформенные — платформенной команде.

### Где деньги

Структура команд — это **инвестиция, которую ты не можешь поменять быстро**.
Найм и формирование команды занимают месяцы, изменение оргструктуры — quarters.

Если ты планируешь архитектуру, которая требует другой структуры команд —
предупреди руководство **заранее**. «Чтобы сделать микросервисы, нам нужны 4
команды, не одна» — это разговор HR + найм + квартальное планирование, не
«сейчас решим».

Книги для углубления:
- *Team Topologies* (Matthew Skelton, Manuel Pais) — про оптимальные структуры
  команд и их связь с архитектурой
- Conway's original paper (1968) — короткий, читается за 20 минут, must-read
