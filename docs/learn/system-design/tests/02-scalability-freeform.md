  - id: sd-q7
    text: |
      Сервис принимает 500 RPS, p99=200ms. Бизнес планирует рост до 5000 RPS
      в течение полугода. Опиши минимум три направления, в которых ты будешь
      искать узкое место, и какие изменения в архитектуре могут понадобиться.
      Обоснуй, почему именно эти направления приоритетны.
    rubric: |
      Сильный ответ упоминает (минимум 3 из):
      - Применение Little's Law: при 500 RPS × 0.2s = 100 одновременных запросов
        в полёте; при 5000 RPS — 1000. Это вопрос о размере пула потоков, коннекшенов,
        worker'ов.
      - Профилирование текущего состояния: где сейчас bottleneck — CPU, IO, БД,
        внешние вызовы, сериализация. «Не оптимизируй вслепую».
      - Stateless-инстансы и горизонтальное масштабирование: можно ли поднять N
        копий сервиса за балансером с автоскейлингом по метрикам.
      - БД как наиболее частый bottleneck: read-replicas, шардинг, кеширование,
        анализ медленных запросов и индексов.
      - Кеширование на разных уровнях: какие запросы повторяются, можно ли вынести
        в Redis/CDN; cache stampede и его митигация.
      - Очереди для асинхронной обработки тяжёлых операций (отправка писем,
        фоновые задачи) — снимает нагрузку с синхронного пути.
      - Метрики, нагрузочное тестирование как способ верификации гипотез о
        bottleneck'е до изменений.

      Слабый ответ — общие слова «добавим серверов» без понимания, где именно
      проблема, без упоминания измерений или trade-off'ов.

      Минимум для прохождения: упомянуто ТРИ осмысленных направления с
      пониманием trade-off'ов и хотя бы упоминание необходимости измерять перед
      оптимизацией.
    remediation: sd-q7
  - id: sd-q8
    text: |
      Объясни, почему consistent hashing предпочтительнее простого hash-based
      шардинга (`hash(key) % N`) при необходимости динамически добавлять и
      удалять ноды в кластере. Опиши механику и упомяни роль virtual nodes.
    rubric: |
      Сильный ответ:
      - При `hash(key) % N` изменение N меняет распределение почти всех ключей —
        нужна полная rehash и перенос ~(N-1)/N данных. На большом кластере это
        фактически downtime.
      - Consistent hashing использует кольцо хешей (обычно 2^32 или 2^64): ноды
        и ключи мапятся в одно пространство; ключ принадлежит «следующей по кольцу»
        ноде по часовой стрелке.
      - Добавление/удаление ноды затрагивает только ~1/N данных — соседей по
        кольцу. Перенос данных локализован.
      - Virtual nodes (vnodes) — каждая физическая нода представлена N точками
        на кольце (обычно 100-200). Это (1) выравнивает распределение при разном
        размере нод и (2) при отказе ноды нагрузка распределяется по многим
        соседям, а не падает на одного.
      - Используется в Cassandra, DynamoDB, Riak, Memcached с consistent hashing
        клиентом.

      Слабый ответ — путает hash-based с consistent или говорит «consistent
      лучше» без объяснения механики кольца и локальности перераспределения.

      Минимум для прохождения: понимание, что consistent hashing минимизирует
      переносы данных при изменении N, плюс хотя бы поверхностное упоминание
      vnodes или кольцевой структуры.
    remediation: sd-q8
