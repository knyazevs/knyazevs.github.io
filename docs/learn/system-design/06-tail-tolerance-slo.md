# Tail tolerance, SLO и graceful degradation

Сделать систему надёжной на пиках нагрузки и при сбоях downstream'а — это не
«больше железа», а правильные паттерны взаимодействия. Здесь — про SLO как
инструмент управления, latency budget'ы и подходы к graceful degradation.

## SLI, SLO, SLA, error budgets

### Три похожих термина

**SLI** (Service Level Indicator) — конкретная метрика. Например, «p99 latency
endpoint /api/checkout».

**SLO** (Service Level Objective) — внутренняя цель команды по SLI. Например,
«p99 < 500ms в 99.9% времени». Это **обещание себе**.

**SLA** (Service Level Agreement) — обещание клиенту с компенсацией за нарушение.
Например, «99.9% uptime, иначе 10% от месячной платы возвращается». Это
**обещание другим**.

Важное правило: **SLA должен быть слабее SLO**. Если SLO = 99.9% и SLA = 99.9% — у
тебя нет запаса. Любое отклонение от SLO означает нарушение SLA. Лучше SLO = 99.95%,
SLA = 99.9% — можно «недотянуть» до SLO, но не нарушить SLA.

### Error budget

Если у тебя SLO = 99.9% — значит, разрешено 0.1% времени не отвечать корректно. За
месяц это 43 минуты. Это **твой error budget**.

Зачем это нужно как инструмент:

- **Решения о рискованных деплоях.** Бюджет потрачен на 80% к середине месяца?
  Замораживаем нерискованные релизы. Бюджет почти не тронут? Можно деплоить
  эксперименты.
- **Балансирование скорости и надёжности.** Команда продукта хочет чаще релизить —
  это съедает error budget. Команда инфраструктуры хочет тратить время на
  надёжность — снижает использование. Error budget — общая валюта переговоров.
- **Отказ от gold-plating.** Если ты не используешь свой error budget — значит,
  инвестируешь в надёжность больше, чем нужно. Можно перенаправить ресурсы.

Концепция из Google SRE Book — must-read, если работаешь с distributed services.

### На практике

Что делать:

- **Поставить SLO для всех user-facing эндпоинтов.** Без числа — нет управления.
- **Дашборд error budget'а** на видном месте. Команда должна видеть, сколько
  потратила.
- **Алерты на сжигание budget'а** (burn rate alerts), не на каждое нарушение SLI.
  Один спайк не страшен; устойчиво высокий error rate — страшен. Стандартная
  формула: алерт, если за 1 час сожгли больше, чем должны за 36 часов.
- **Постмортемы для крупных потерь budget'а.** Если за один инцидент потратил
  месячный budget — разбор и action items.

### Когда модель усложняется

SLO на p99 latency — частая ошибка. **p99 — это статистика последних N минут**. В
момент инцидента все запросы могут быть медленными, но p99 за всю неделю может не
сдвинуться, если инцидент короткий. Решение: алертить по короткому окну (5-15
минут p99), не по дневному.

Также не путай **availability SLO** (% запросов с success) с **latency SLO** (% в
пределах X ms). Это разные метрики, у них разные error budget'ы.

## Latency budget и deadline propagation

### Откуда берётся budget

Endpoint имеет SLO `p99 < 500ms`. Это твой **общий latency budget** на пользовательский
запрос.

Запрос проходит через несколько звеньев:

1. API gateway: routing, auth — 50ms
2. Backend service: бизнес-логика + 5 параллельных backend-вызовов
3. Каждый backend-вызов: своя обработка + БД-запрос

Чтобы уложиться в 500ms суммарно, надо **распределить budget между звеньями**:

- Gateway: 50ms
- Backend service: 400ms (включает время worst-of параллельных вызовов)
- Каждый backend-вызов: 350ms (worst case)
- БД-запрос внутри: 200ms

Это **latency budget breakdown**. Без него команды строят свои SLO независимо, и
сумма получается больше total budget'а.

### Deadline propagation

В каждый downstream-вызов передаётся **deadline** — момент времени, до которого он
должен ответить.

Gateway получил запрос в 10:00:00.000 с budget 500ms → дедлайн 10:00:00.500.

Gateway тратит 50ms, передаёт downstream'у дедлайн 10:00:00.500 (или относительный
— «осталось 450ms»).

Downstream получает дедлайн, проверяет — успеет ли. Если очевидно нет (например,
осталось 50ms, а БД-запрос обычно 200ms) — возвращает 504 сразу, **не делая
бесполезной работы**.

Что это даёт:

- Гарантия, что весь путь укладывается в SLO.
- **Fail fast** — не тратить ресурсы на гарантированно бесполезную работу.
- Защита от каскадного timeout: один медленный downstream не «съедает» время,
  отведённое для retry.

В **gRPC это first-class механизм** (`grpc.WithTimeout` распространяется по цепочке
вызовов автоматически). В REST приходится передавать через header'ы
(`X-Request-Deadline` или подобный) и реализовывать обработку самому.

### На практике

Типичная история: добавили retry на downstream-вызовы, чтобы повысить надёжность.
Каждый retry без учёта deadline'а удваивает worst-case latency. p99 сервиса уезжает,
а команда не понимает почему.

Решение: retry **с budget'ом** — если осталось меньше X — не retry'им. Или общий
retry budget на запрос — не больше N retry'ев суммарно по всем вызовам.

### Когда модель усложняется

Распределение budget'а между параллельными вызовами — отдельная задача. Если у тебя
5 параллельных backend-вызовов с одинаковым deadline'ом — общая latency = max из
них. Если один очевидно медленнее остальных — он становится bottleneck'ом, и его
дедлайн надо ставить с учётом этого.

Reformulation: budget — это не «равномерно распределить», а «дать каждому столько,
сколько ему нужно, плюс резерв на остальные».

## Graceful degradation и load shedding

### Идея

Под перегрузкой или при сбое downstream'а вместо «падать целиком» — **деградировать
частично**, оставляя работающим самое важное.

### Способы

**1. Feature flags для тяжёлых функций.** Под нагрузкой можно отключить
рекомендации, аналитику, персонализацию — оставить базовый функционал. Юзер видит
«упрощённую» версию страницы, но видит её. Switch обычно глобальный (включил/выключил
функцию для всех), управляемый из админки или auto-trigger по метрикам.

**2. Опциональные backend-вызовы с fallback.** Если recommendations service не
ответил за 200ms — показываем страницу без рекомендаций. Если cache miss и БД
медленная — показываем устаревшие данные из кеша. Каждый downstream-вызов
проектируется с вопросом «что показать, если он не ответил?».

**3. Circuit breaker.** После N подряд ошибок от downstream — перестаём звонить
туда на T секунд, возвращаем fallback. Защищает downstream от добивания (он и так
плохо себя чувствует) и нас от каскадного timeout'а. Через T секунд — пробное
открытие (half-open state), один тестовый запрос, если успешно — возобновляем.

**4. Load shedding by class.** Под перегрузкой дропаем запросы по приоритету:

- Гость → 503, оплачивающий пользователь → обслуживаем
- Аналитика/batch → откладываем, real-time запросы → обслуживаем
- API rate limits снижаем для бесплатных тиров, премиум не задеваем
- Bot-traffic дропается первым

**5. Read-only mode под deep failure.** Если основная БД упала, переходим в
read-only с реплики — пользователи видят данные, но не могут писать. Лучше, чем
полный outage. Для многих сценариев (новостной сайт, документация) это достаточно.

### На практике

Что делать:

- **Спроектировать каждый downstream-вызов с вопросом «что показать, если он не
  ответит?».** Если ответ «упасть целиком» — это уязвимость. Лучше иметь fallback
  на каждый.
- **Circuit breaker — не опция.** Любая интеграция с downstream сервисом без
  circuit breaker — это бомба замедленного действия. Под нагрузкой downstream
  ляжет, ты будешь упорно ему звонить и добивать. Готовые библиотеки: Resilience4j
  (Java), Polly (.NET), gobreaker (Go).
- **Тестировать degraded mode** в chaos engineering — целенаправленно убивать
  downstream и смотреть, действительно ли всё работает с fallback'ами. Иначе
  fallback'и есть только в коде, но не в реальности.
- **Документировать graceful degradation.** Поддержка должна знать: «если видишь
  что рекомендации не показываются — это ожидаемое поведение под нагрузкой, не баг».

### Где деньги

Graceful degradation — это **инвестиция в код**, не в железо. Каждый fallback
требует продумывания, тестирования, поддержки. Но один работающий fallback окупается
за один инцидент, который не превратился в outage.

Грубое правило: если downstream-вызов на критическом пути — у него **должен быть**
план B. Если нет плана B — переноси работу с критического пути (асинхронно или
вообще убирай).
