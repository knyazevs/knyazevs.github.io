# Время в распределённой системе

«Который час?» — самый сложный вопрос в распределённых системах. От ответа зависит,
как упорядочиваются события, как разрешаются конфликты, как работают распределённые
транзакции.

## Почему «который час» — сложный вопрос

### Физические часы расходятся

У каждого сервера свои часы, основанные на кварцевом осцилляторе. Кварц нестабилен
— часы дрейфуют (это явление называется clock drift, «дрейф часов») со скоростью
~10ms в день. Без коррекции через неделю часы двух серверов разойдутся на ~70ms.

NTP (Network Time Protocol) синхронизирует часы по сети с внешним источником
времени. Точность:

- В пределах одного датацентра: ~1-10ms
- Между датацентрами: ~10-100ms
- Через интернет: до 100ms+

Этого **недостаточно** для упорядочивания событий, особенно если события происходят
на разных серверах в разных регионах. Если запись с timestamp 10:00:00.005
произошла «после» записи с timestamp 10:00:00.003 — это может быть неправда,
потому что часы расходятся на 10ms и больше.

### Зачем нужно ordering

Распределённые БД, чаты, коллаборативные редакторы — все нуждаются в ответе на
вопрос «что случилось раньше». От этого зависит:

- **Last-write-wins resolution** (LWW, разрешение конфликтов через «побеждает
  запись с большим timestamp'ом») — какой write «победил» при конфликте
- **Causal ordering** (причинно-следственный порядок) — отображение событий в
  правильном порядке
- **Snapshot isolation** (изоляция снимков — каждая транзакция видит свой снимок
  данных на момент начала) в распределённой БД — какая версия данных «текущая»
- **Distributed locks** (распределённые блокировки) — кто захватил lock первым

Без надёжного ordering эти механизмы не работают корректно.

### На практике

Что делать:

- **Не полагайся на физические timestamp'ы для критических решений.** Если у тебя
  «победит запись с большим timestamp'ом» — два сервера с расхождением часов могут
  потерять данные.
- **Используй логические часы для ordering** (см. ниже).
- **Принимай, что NTP иногда отказывает.** Известны баги с leap second'ами
  («секунда координации» — добавляется раз в несколько лет к UTC для синхронизации
  с астрономическим временем; для системных часов это неожиданный «прыжок назад»
  на одну секунду), которые клали целые сервисы (например, инцидент Cloudflare
  2017 года из-за обработки отрицательного интервала после leap second'а).

## Lamport, vector clocks, HLC — что и зачем

### Lamport clocks (1978)

Идея: вместо физического времени каждое событие получает **логический счётчик**.

Правила:

- Локальное событие → инкремент счётчика
- При отправке сообщения → отсылается значение счётчика
- При получении сообщения → счётчик = `max(local, received) + 1`

Гарантирует: если событие A causally произошло до B (A → B), то L(A) < L(B).

**Не гарантирует обратное**: L(A) < L(B) не значит, что A → B. Может быть, A и B
конкурентны (происходили независимо).

Lamport clocks дают **partial ordering** — частичный порядок. Достаточно для
базовых задач (например, какая версия документа новее в одной цепочке правок).

### Vector clocks

Расширение Lamport: каждый узел держит вектор счётчиков [N] — по одному на каждую
ноду в кластере.

Правила:

- Локальное событие → инкремент **своего** счётчика
- При отправке → отсылается весь вектор
- При получении → поэлементный max + инкремент своего

Позволяет различить **causal vs concurrent**:

- Если все элементы V(A) ≤ V(B) → A → B (причинная связь)
- Иначе — события конкурентны

Это критично для разрешения конфликтов в multi-leader репликации (мульти-мастер:
writes принимаются на нескольких нодах одновременно, и при возврате связи нужно
понять, какой write случился раньше). Используется в Dynamo paper (Amazon, 2007),
Riak.

Цена: размер вектора растёт с количеством нод, передаётся в каждом сообщении. Для
кластера из 100 нод — 100 счётчиков на каждое сообщение, что уже много.

**Cassandra изначально использовала vector clocks, в версии 1.0 убрала их в пользу
last-write-wins с физическим timestamp'ом** — оказалось проще и для большинства
use case'ов достаточно. Поучительно: правильное теоретически решение проиграло
практически приемлемому.

### Hybrid Logical Clocks (HLC)

Гибрид физического и логического времени: timestamp = `(physical_time, logical_counter)`.

Правила генерации timestamp'а:

- Берём текущее физическое время
- Если оно больше последнего HLC — сбрасываем логический счётчик в 0
- Если равно или меньше (clock drift или conflict) — инкрементируем логический счётчик

Свойства:

- **Близко к физическому времени** — можно использовать для TTL и для отображения
  юзеру (timestamp выглядит как нормальное время)
- **Защищён от clock drift** (дрейфа часов) через логическую часть (счётчик
  растёт даже если физические часы временно «откатились»)
- **Размер фиксированный**, не растёт с числом нод (в отличие от vector clocks)

Используется в **CockroachDB, YugabyteDB, MongoDB (4.0+)** — стал стандартом для
современных распределённых БД. Хорошо балансирует «достаточная точность ordering'а
+ не очень дорого + удобно».

### На практике

Когда что использовать:

- **Lamport clocks** — почти никогда напрямую. Концептуальная база.
- **Vector clocks** — если нужно явно различать concurrent vs causal операции и
  нод немного (десятки). Реже встречается в современных системах.
- **HLC** — выбор по умолчанию для современной распределённой БД. Решает 90%
  практических задач ordering'а.
- **TrueTime-style** (см. ниже) — если ты Google и есть атомные часы. Иначе —
  недоступно.

## TrueTime — Spanner и инфраструктурный обход

### В чём идея

Уникальный подход Google. В каждом датацентре стоят атомные часы и GPS-приёмники.
API `TT.now()` возвращает не точное время, а **интервал** `[earliest, latest]` с
гарантией, что реальное время находится в нём (обычно ширина интервала ε ~7ms).

При commit'е транзакции Spanner:

1. Берёт `now() = [earliest, latest]`
2. Использует `latest` как timestamp commit'а
3. **Ждёт** до момента `latest` (commit wait, обычно 1-7ms)

После commit wait любая последующая транзакция на любом сервере получит timestamp
**строго больше**, потому что её `now()` будет уже после `latest` предыдущей. Так
Spanner получает линеаризуемость без распределённого консенсуса по времени для
каждого commit'а.

### Что это даёт

- Линеаризуемая глобальная распределённая БД с приемлемой задержкой
- Не нужен Paxos/2PC (two-phase commit, двухфазная фиксация распределённой
  транзакции) для каждой read-операции — достаточно прочитать с любой
  реплики на правильном snapshot timestamp
- Read-only транзакции вообще без блокировок

### Где это работает

Только в Google. Атомные часы в каждом ДЦ — инфраструктурная инвестиция, доступная
единицам.

AWS Time Sync Service приближается к этому (использует satellite-synced clocks),
но без формальных гарантий ε. AWS не предоставляет публично гарантию «реальное
время точно в интервале X».

CockroachDB и Yugabyte используют HLC как «приближение TrueTime для бедных» — без
атомных часов они не могут давать жёсткой гарантии ε, но для большинства задач
достаточно.

### Урок

TrueTime — пример того, как инфраструктурная инвестиция может убрать алгоритмическую
сложность. Большинство распределённых БД делают сложные алгоритмы (Paxos, vector
clocks) потому, что не могут позволить себе атомные часы. Google смог — и получил
более простую модель.

Для архитектора важен не сам TrueTime, а паттерн: **иногда правильный ход — не
«улучшить алгоритм», а «улучшить инфраструктуру под более простой алгоритм»**.
Доступно не всем, но легитимный подход.

### Когда модель усложняется

TrueTime даёт линеаризуемость **внутри Spanner**. За пределами — обычные проблемы
распределённого времени. Если у тебя приложение делает операции в Spanner и в
Redis параллельно, ordering между ними не гарантирован.

И ε не равно нулю. Если две операции произошли в окно ε (~7ms), их относительный
порядок не определён — Spanner просто использует логические тайбрейкеры. Для
человеческой шкалы это мгновение, для high-frequency trading — вечность.
