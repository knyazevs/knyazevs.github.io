# Little's Law

## Формула

`L = λ × W`, где:

- **L** — среднее число запросов «в полёте» (concurrency, work-in-progress)
- **λ** — throughput (requests per second)
- **W** — average residence time (≈ latency)

## Зачем это на практике

Применима к **любой стабильной системе** с потоком запросов — не зависит от
распределений, не требует предположений. Используется для **сайзинга**:

- API держит 1000 RPS при средней latency 100ms → одновременно в системе ~100
  запросов. Пул потоков / коннекшенов / воркеров должен быть **минимум** 100
  (с запасом — 150-200).
- Если пул меньше — формируется очередь, latency растёт, в худшем случае timeout cascade.
- Если значительно больше — ресурсы простаивают, но это безопаснее, чем недосайзить.

## Чего НЕ описывает

- Не говорит, **как** уменьшить latency или увеличить throughput.
- Не учитывает peak load — только steady state. На спайках (например, утренний
  пик у новостного сайта) пул должен быть рассчитан на peak, не average.
- Не работает на нестабильных системах (постоянно растущая очередь — система
  уже не в стабильном состоянии).

## Типичный диалог на интервью

> — У нас 200 RPS на API, средняя latency 50ms. Какой нужен пул HTTP-клиентов?
>
> — По Little's Law concurrency = 200 × 0.05 = 10. Минимум 10 коннекшенов;
> с запасом на p99 latency и спайки — 30-50.

Знание Little's Law — это buying credibility за 30 секунд.

## Связь с другими понятиями

- Если throughput фиксирован, а latency растёт — растёт concurrency, а значит
  потребление памяти и переключение контекста. Это путь к деградации.
- Если хочешь поднять throughput при том же пуле — единственный способ снизить
  latency (W). Это переводит разговор в плоскость профилирования.
