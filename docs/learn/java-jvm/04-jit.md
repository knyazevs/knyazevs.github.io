# 04. JIT: от байткода до машинного кода

JIT (Just-In-Time) компилятор — это причина, по которой Java-сервис через
несколько минут после запуска начинает работать быстрее, чем в первые секунды.
Понимание JIT критично для Tech Lead: это объясняет warmup-паузы, неожиданные
performance cliff'ы и то, почему некоторые «оптимизации» кода на самом деле
мешают JIT делать свою работу.

---

## 1. Путь от байткода к машинному коду

### Теория

JVM исполняет байткод (промежуточное представление `.class`-файлов) тремя
способами, последовательно переключаясь по мере накопления профиля.

**Интерпретатор (Level 0)** читает и выполняет байткод инструкция за инструкцией.
Медленно: каждая операция проходит через switch-диспатч в C++-коде JVM. Зато
запускается немедленно — нет никакой паузы на компиляцию. Параллельно
интерпретатор собирает профиль: считает вызовы методов, итерации циклов,
фиксирует реальные типы значений в полиморфных вызовах.

**C1 (Client Compiler, Level 1–3)** — быстрый компилятор с базовыми
оптимизациями. Компилирует быстро, генерирует неплохой машинный код, продолжает
собирать профиль (на уровнях 2 и 3).

**C2 (Server Compiler, Level 4)** — агрессивный оптимизирующий компилятор.
Использует накопленный профиль, применяет спекулятивные оптимизации, генерирует
близкий к рукописному C++ машинный код. Компилирует долго — десятки миллисекунд
на метод — но делает это в фоновом потоке.

**Tiered Compilation** (включена по умолчанию с Java 8) объединяет всё это в
единый пайплайн:

```
Level 0: интерпретатор                    ← все методы стартуют здесь
Level 1: C1 без профилирования
Level 2: C1 с лёгким профилированием
Level 3: C1 с полным профилированием      ← горячий метод ждёт очереди C2
Level 4: C2 на основе профиля Level 3     ← пиковая производительность
```

Горячий метод проходит путь 0→3→4. Метод, который вызывается редко, может
остаться на Level 2 навсегда — C2-компиляция для него просто невыгодна.

Порог компиляции без tiered: `-XX:CompileThreshold=10000` (10 тысяч вызовов до
C2-компиляции). С tiered — JVM управляет этим сам через эвристики, учитывая
размер очереди компиляции и частоту вызовов.

### На практике

Это и есть реальное объяснение «прогрева» (warmup). Первые N тысяч запросов
обрабатываются интерпретатором или C1. Latency выше, throughput ниже. После того
как горячие пути прогреваются до Level 4 — характеристики стабилизируются.

В production это означает: после деплоя не переключай 100% трафика сразу.
Используй canary-деплой: небольшая доля трафика прогревает JIT, потом
переключаешь остальных. Иначе получишь spike latency в момент переключения.

**Решение в лоб — запустить нагрузочный тест перед деплоем** — частично
работает для функциональной корректности, но не всегда воспроизводит production
профиль. JIT профилирует конкретные типы данных и ветки: синтетическая нагрузка
может прогреть не те пути.

## 2. Ключевые оптимизации C2

### Теория

**Inlining** — важнейшая оптимизация, основа для всего остального. Вместо вызова
метода JIT буквально вставляет его тело в место вызова. Зачем? Вызов метода в
JVM — это:
- push аргументов на стек
- создание нового stack frame
- переход по адресу метода
- возврат значения и pop frame

Inlining убирает весь этот overhead. Но главнее другое: большинство оптимизаций
(constant folding, dead code elimination, escape analysis) работают в пределах
одного метода. Inlining расширяет эти пределы — JIT видит больший контекст и
оптимизирует агрессивнее.

Ограничения: метод длиннее 35 байткод-инструкций не инлайнится по умолчанию
(`-XX:MaxInlineSize=35`). Маленькие геттеры, сеттеры, делегирующие методы — все
инлайнятся. Большие бизнес-методы — нет. Это одна из причин, почему «паттерн
тонких методов» в Java — не просто стиль, а производительность.

**Devirtualization и спекулятивный inlining**: JVM знает из профиля, что
`collection.add()` в 99.9% случаев это `ArrayList.add()`. C2 вставляет:
```
if (collection instanceof ArrayList) {
    // inlined ArrayList.add()  ← быстрый путь
} else {
    uncommon_trap();  ← редкий путь → deoptimization
}
```
Это спекулятивная оптимизация: JIT делает ставку на профиль. Если ставка
неверна — uncommon trap срабатывает и запускается deoptimization (подробно в
следующей секции).

**Loop-оптимизации**: unrolling (разворачивание цикла — меньше итераций, больше
работы в каждой), vectorization (автоматическое использование SIMD-инструкций
процессора для обработки нескольких элементов за такт), loop invariant hoisting
(вынос вычислений, не зависящих от итерации, за тело цикла).

**Intrinsics**: для ряда стандартных методов JIT не генерирует код вовсе — он
знает оптимальную нативную реализацию и вставляет её напрямую.
`System.arraycopy`, `Math.sqrt`, `String.equals`, методы `Unsafe`, `Arrays.fill`
— всё это intrinsics. Поэтому `System.arraycopy` быстрее ручного цикла
копирования: JIT вставляет `memcpy` или векторный ассемблер, а не генерирует
байткод-цикл.

### На практике

**Мегаморфный callsite** — термин для callsite, через который проходят более 2
конкретных типов. JIT отказывается от девиртуализации и инлайнинга для таких
точек: слишком много вариантов, guard не окупается. Вызов уходит через vtable
(таблица виртуальных методов — структура в памяти объекта, содержащая указатели
на методы). Это медленнее в разы.

Реальный паттерн: у тебя есть `List<Handler> handlers`, все они долгое время
были `FooHandler`. Потом добавили `BarHandler`. Throughput упал. Это не утечка,
не GC — это callsite стал бимоморфным. Добавили третий тип — стал мегаморфным,
инлайнинг исчез, производительность упала ещё.

**Решение в лоб — сделать методы final** — работает, но только для прямых
вызовов. `final` метод инлайнится лучше: JVM точно знает, что реализация
единственная, guard не нужен. Но интерфейсные вызовы `final` не убирает.
Настоящее решение — минимизировать полиморфизм в горячих путях: либо один тип в
коллекции, либо явный dispatch вместо полиморфизма.

## 3. Deoptimization — когда предположения нарушаются

### Теория

Спекулятивные оптимизации C2 — это ставки. JIT делает допущение: «этот callsite
всегда будет `ArrayList`». Если допущение нарушается в runtime, нужен откат.

Механизм deoptimization:

1. JIT вставляет **uncommon trap** — специальную точку входа в «нетипичный» путь
2. При срабатывании trap: текущий поток приостанавливается
3. Стек «разворачивается» (deopt) — JVM восстанавливает состояние интерпретатора
4. Метод помечается `deoptimized`, интерпретатор продолжает с нужной точки
5. Со временем JIT перекомпилирует метод с обновлённым профилем

Это не катастрофа — отдельный deopt незаметен. Проблема возникает, когда deopt
происходит систематически: callsite регулярно «промахивается», метод циклически
компилируется и деоптимизируется.

**OSR (On-Stack Replacement)** — компиляция горячего цикла прямо во время его
исполнения, без выхода из метода. Нужен когда метод вызывается один раз, но
внутри него цикл с миллионами итераций. JIT замечает горячий цикл, компилирует
«горячий» фрагмент и переключает исполнение на скомпилированную версию прямо в
середине итерации. В `-XX:+PrintCompilation` такая компиляция видна с суффиксом
`%`.

**Compilation bail-out**: JIT начал компилировать метод, но обнаружил паттерн,
который не умеет оптимизировать — например, exception handler в горячем пути или
слишком сложный граф потока управления. Откатывается к C1 или интерпретатору.
Это тоже видно в PrintCompilation как `not compiled`.

### На практике

Типичный **performance cliff**: сервис работал стабильно, потом появился новый
тип объекта (новая реализация интерфейса). Throughput упал на 20–30%. Никаких
изменений в бизнес-логике.

Что произошло: callsite стал мегаморфным, C2 снял инлайнинг, начались deopt'ы
для методов, которые были скомпилированы под предыдущий профиль. JIT
перекомпилирует их с новым профилем, но время уже потеряно — и новый profile без
девиртуализации.

Это не баг — это поведение speculative JIT. Диагностика: смотри на `-XX:+PrintCompilation`
на частые перекомпиляции одних и тех же методов.

**Решение в лоб — перезапустить JVM после добавления нового типа** — работает как
workaround, но не решает причину. Если новый тип будет появляться регулярно —
проблема вернётся. Настоящее решение: изолировать полиморфный callsite в отдельный
«диспатч» слой и держать горячий путь мономорфным.

## 4. GraalVM как JIT через JVMCI

### Теория

**JVMCI (JVM Compiler Interface)**, появившийся в Java 9, позволяет заменить
встроенный JIT-компилятор на внешний — и при этом сам компилятор может быть
написан на Java. Graal Compiler — именно такая реализация: JIT на Java, который
заменяет C2.

Зачем это нужно:

- **Truffle** — фреймворк для написания интерпретаторов поверх JVM. JavaScript
  (GraalJS), Python (GraalPy), Ruby (TruffleRuby) исполняются как Truffle-программы,
  которые Graal JIT оптимизирует специфичным образом (partial evaluation).
- **GraalVM native-image** — AOT-компиляция в нативный бинарник. Здесь JIT вообще
  нет: весь код компилируется заранее, нет warmup, мгновенный старт. Другой
  trade-off: нет профиля, нет спекулятивных оптимизаций под реальную нагрузку.
- В редких случаях Graal JIT даёт лучший throughput чем C2 на специфичных
  workloads благодаря более точному анализу escape.

### На практике

В production по умолчанию работает HotSpot C2, не Graal. Переключиться:
`-XX:+UseJVMCICompiler`. Это экспериментально для большинства production
workloads — характер оптимизаций отличается, нужно профилировать конкретно.

Graal native-image — другая история. Активно используется в Quarkus, Micronaut,
Spring Boot 3.x для serverless и CLI-инструментов, где время запуска критично.
Потеря: без JIT нет адаптации под runtime-профиль, пиковый throughput обычно
ниже чем у прогретого HotSpot.

## 5. Диагностика JIT

### Теория

**`-XX:+PrintCompilation`** — самый доступный инструмент. Каждая компиляция
печатает строку:

```
  1  23    3       java.lang.String::equals (68 bytes)
```

Поля по порядку: временная метка (мс с запуска), compile id, уровень компиляции
(1–4), имя метода, размер байткода.

Специальные маркеры:
- `%` — OSR-компиляция (On-Stack Replacement, см. выше)
- `!` — метод содержит exception handler
- `n` — native-обёртка (intrinsic или JNI)
- `s` — синхронизированный метод
- `made not entrant` — метод деоптимизирован, новые вызовы идут через
  интерпретатор
- `made zombie` — метод вытеснен из code cache (давно деоптимизирован, место
  нужно для новых компиляций)

**`-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining`** добавляет дерево
инлайнинга под каждой компиляцией. Ищи:
- `too big` — метод превысил MaxInlineSize
- `not inlineable` — native или abstract без конкретной impl
- `megamorphic` — JIT отказался от девиртуализации

**JITWatch** — GUI-инструмент для анализа compilation log. Показывает triview:
исходник → байткод → сгенерированный ассемблер. Помогает увидеть, что именно JIT
сделал с конкретным методом: где инлайнинг сработал, где нет, что стало
векторизовано.

**`-XX:+PrintAssembly`** (требует `hsdis` — дизассемблер HotSpot, ставится
отдельно) выводит ассемблер сгенерированного кода. Нужен, когда хочешь проверить
наличие SIMD-инструкций (ymm/xmm регистры = AVX/SSE векторизация) или убедиться,
что конкретная оптимизация применилась.

### На практике

Типичный сценарий диагностики performance-проблемы:

1. Запусти с `-XX:+PrintCompilation > compile.log 2>&1`
2. Найди методы, которые перекомпилировались много раз (много строк с одним именем)
3. Найди `made not entrant` на горячих методах — это deopt
4. Добавь `-XX:+PrintInlining` и посмотри почему не инлайнится то, что должно
5. Используй JITWatch для визуализации если нужна детализация

**Решение в лоб — увеличить MaxInlineSize** — иногда помогает для конкретного
метода, но увеличивает давление на code cache и время компиляции. Правильнее
рефакторить методы: разбить большой метод на более мелкие горячие пути, которые
укладываются в порог.

## 6. Ускорение прогрева в production

### Теория

Warmup — это задержка между деплоем и пиковой производительностью. Для
высоконагруженных сервисов это реальная проблема: несколько минут degraded
latency после каждого деплоя.

Инструменты:

**CRaC (Coordinated Restore at Checkpoint)** — механизм OpenJDK (production-grade
с Java 21). Идея: запускаешь JVM, прогреваешь до Level 4 по всем горячим путям,
делаешь checkpoint (снимок состояния процесса). При следующем запуске — restore из
checkpoint'а. JVM стартует с уже прогретым JIT, без warmup-паузы. Spring Boot
3.2+ поддерживает CRaC из коробки.

**Class Data Sharing (AppCDS)** — сохраняет загруженные классы в shared archive.
При следующем запуске классы загружаются из архива без парсинга `.class`-файлов.
Ускоряет startup (меньше работы classloader'а), но это не JIT: код всё равно
начинает с интерпретатора.

**GraalVM native-image** — убирает warmup полностью, компилируя всё заранее.
Подходит для функций с коротким lifetime (serverless, CLI). Для long-running
сервисов под высокой нагрузкой прогретый HotSpot обычно выигрывает по throughput.

### На практике

Canary deployment + health-based traffic shifting — минимальная защита. Деплоишь
на одну инстанцию, ждёшь пока latency-метрики стабилизируются (обычно 2–5 минут
при реальном трафике), потом переключаешь остальные.

CRaC даёт лучший результат но требует аккуратности: в момент checkpoint'а нельзя
иметь открытых соединений к БД, socket'ов. Spring Boot 3.2 добавил `LifecycleBean`
интеграцию — закрывает и восстанавливает соединения автоматически.

**Решение в лоб — держать «горячий» standby процесс** — работает, но дорого:
удваиваешь ресурсы только ради warmup. Оправдано только для критически важных
сервисов где стоимость degraded latency высока.

## Каверзные вопросы к интервью

**Почему `final` метод инлайнится лучше?**  
JVM точно знает, что реализация единственная — не нужен guard check на тип.
Обычный виртуальный метод требует проверки перед инлайнингом. `final` — нет.

**Что такое мегаморфный callsite и почему он убивает производительность?**  
Callsite с более чем 2 конкретными типами в профиле. JIT не может сделать
девиртуализацию (слишком много вариантов для guard). Вызов уходит через vtable
без инлайнинга. Нет inlining → нет constant folding, нет escape analysis, нет
последующих оптимизаций. Throughput падает в разы.

**Как JIT знает что метод горячий?**  
Интерпретатор поддерживает счётчики вызовов и итераций цикла прямо в bytecode.
При превышении порога метод ставится в очередь компиляции. С tiered — решение
принимается по эвристике с учётом текущей нагрузки на compiler thread'ы.

**Почему JMH использует `Blackhole`?**  
Без Blackhole JIT видит, что результат вычисления нигде не используется, и
удаляет его как dead code (dead code elimination). Benchmark измеряет нулевую
работу. `Blackhole.consume()` — специальный метод, который JIT не может исключить:
он «потребляет» результат и предотвращает устранение.

**Что такое compilation bail-out?**  
JIT начал компиляцию метода, но наткнулся на паттерн, который не умеет
оптимизировать: сложный exception handler в горячем пути, нестандартная структура
байткода, превышение сложности. Откатывается к C1 или интерпретатору. Метод
остаётся на более низком уровне навсегда (до следующего деплоя).

**Почему `System.arraycopy` быстрее ручного цикла?**  
Это intrinsic: JIT не генерирует код для него, а вставляет оптимальный нативный
вариант (memcpy или векторный ассемблер). Ручной цикл JIT компилирует из
байткода, результат — хуже чем специально написанный C++/asm.
