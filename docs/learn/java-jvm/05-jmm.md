# Java Memory Model: happens-before, volatile, final, reordering

## 1. Зачем нужна модель памяти

Интуитивно кажется, что если ты записал переменную, другой поток её сразу увидит. В однопроцессорном мире 1990-х это было примерно правдой. Сегодня — нет.

Современный CPU не работает напрямую с RAM. Между ядром и оперативной памятью три уровня кэша: L1 (единицы МБ, ~4 нс), L2 (~256 КБ, ~12 нс), L3 (десятки МБ, ~40 нс) — и только потом DRAM (~60–80 нс). Когда поток A на ядре 0 пишет в переменную, она сначала оседает в store buffer этого ядра. Поток B на ядре 1 читает из своего кэша. Когда store buffer сбрасывается и cache coherence протокол (MESI) синхронизирует кэши — зависит от архитектуры и от того, встретятся ли барьерные инструкции на пути.

Параллельно с железом работает компилятор. JIT-компилятор и javac вправе переупорядочивать инструкции, если это не меняет результат внутри одного потока. «Внутри одного потока» — ключевое. Между потоками переупорядочивание может стать видимым и ломать логику.

Получается два класса проблем.

**Visibility (видимость)**: поток A записал значение, поток B читает старое из своего кэша. Никакой гонки по времени нет — B честно читает «позже», но видит устаревшее значение.

**Reordering (переупорядочивание)**: поток A делает записи в порядке X→Y, но поток B видит их в порядке Y→X или вовсе видит только одну из них.

**Java Memory Model (JMM)**, специфицированная в JSR-133 (Java 5) и описанная в главе 17 JLS, определяет формальный контракт: при каких условиях действие одного потока гарантированно видно другому. Это не описание того, как работает конкретный CPU — это контракт между программистом и всем стеком (JVM, JIT, CPU). Если программист соблюдает условия JMM, JVM обязана дать гарантии. Если не соблюдает — поведение не определено (data race → UB в терминах JMM).

На практике: баги JMM воспроизводятся на ARM (слабая модель памяти) чаще, чем на x86 (TSO — Total Store Order — сильная модель, где единственный легальный reorder это StoreLoad). Если программа корректна по JMM, она будет корректна на обеих архитектурах.

## 2. Happens-Before

**Happens-before (HB)** — центральное понятие JMM. Формально: если действие X happens-before действия Y, то Y гарантированно видит результаты X. Важно: HB не означает, что X физически произошёл по часам раньше Y. Это гарантия видимости, не порядок по времени.

JMM определяет правила, из которых строятся HB-рёбра.

**1. Program order.** Внутри одного потока все действия упорядочены: любое действие HB следующему по программному порядку. Это не означает, что CPU выполняет их строго последовательно — он может, но результат должен быть такой, как будто выполнил. Для одного потока переупорядочивание не видно.

**2. Monitor lock.** `unlock` на мониторе HB `lock` на том же мониторе, выполненному любым другим потоком. Это основа `synchronized`: выходя из блока, поток публикует все свои записи. Входящий в тот же блок поток гарантированно их увидит.

**3. Volatile write→read.** Запись в `volatile` поле HB любому последующему чтению того же поля из любого потока. «Последующему» — по глобальному порядку записей в volatile, не по часам.

**4. Thread start.** Вызов `Thread.start()` HB любому действию в запущенном потоке. Всё, что ты сделал до `start()`, новый поток увидит.

**5. Thread join.** Любое действие в потоке T HB возврату из `t.join()` в вызывающем потоке. Всё, что поток T сделал до завершения, ты увидишь после `join()`.

**6. Transitivity.** Если X HB Y и Y HB Z, то X HB Z. Это позволяет строить цепочки гарантий.

Что важно понять про HB: он определяет минимум гарантий. Если HB-ребра между двумя действиями нет — поведение не определено. JVM не обязана показывать записи без HB-гарантии, но и не обязана скрывать. Полагаться на «а вот тут JVM всё равно обновит кэш» — data race, даже если на твоей машине не воспроизводится.

**На практике.** Паттерн «флаг остановки»:

```java
// СЛОМАНО без volatile:
boolean running = true;

// Thread A:
running = false;

// Thread B:
while (running) { doWork(); }
```

Поток B может крутиться вечно: JIT имеет право выкинуть чтение `running` из цикла, так как в рамках B никто не меняет `running` — нет HB-ребра между записью A и чтением B. Добавление `volatile` создаёт HB-ребро: volatile write (A) HB volatile read (B). Теперь B гарантированно видит `false`.

## 3. Volatile: видимость и барьеры

`volatile` на поле даёт две гарантии.

**Visibility**: каждая запись в volatile поле немедленно видна всем потокам. Реализуется через flush store buffer + cache invalidation по протоколу когерентности кэша.

**Ordering**: volatile-запись не переупорядочивается с предшествующими обычными записями, volatile-чтение не переупорядочивается с последующими обычными чтениями. На уровне инструкций JVM вставляет memory barrier — на x86 это `MFENCE` или `LOCK XCHG` (StoreLoad barrier), на ARM — `dmb ish` (data memory barrier, inner shareable domain).

StoreLoad barrier — самый дорогой: он опустошает store buffer текущего ядра и инвалидирует чужие кэши. На горячем пути с частыми volatile-записями это заметно, но не катастрофично — порядок одиночной операции в десятки наносекунд.

**Чего volatile НЕ даёт — атомарности составных операций.**

```java
volatile int counter = 0;

// Thread A и Thread B одновременно:
counter++;
```

`counter++` раскладывается в три шага: read → increment → write. Между read и write другой поток может вклиниться и прочитать то же значение. Оба запишут +1, но счётчик вырастет только на 1, а не на 2. Для атомарного инкремента — `AtomicInteger.incrementAndGet()`, который использует CAS (compare-and-swap) инструкцию процессора.

**Volatile long и double.** По JLS, без `volatile` запись/чтение `long` и `double` (64-битных примитивов) не обязаны быть атомарными — JVM может разбить на две 32-битные операции (word tearing). На практике 64-битные JVM делают это атомарно, но спецификация не гарантирует. `volatile long` гарантирует атомарность и по спецификации, и на практике.

## 4. Double-Checked Locking — классическая ловушка JMM

Это учебный пример, по которому задают вопросы на интервью, потому что он выглядит логично и при этом фундаментально сломан без volatile.

```java
// СЛОМАНО (до Java 5 / без volatile):
class Singleton {
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {               // (1) первая проверка — без лока
            synchronized (Singleton.class) {
                if (instance == null) {       // (2) вторая проверка — под локом
                    instance = new Singleton(); // (3) создание
                }
            }
        }
        return instance;
    }
}
```

На первый взгляд логика правильная: если уже создан — без лока, если нет — берём лок и проверяем снова. Проблема в строке (3).

`instance = new Singleton()` компилируется в три шага байткода:
1. Выделить память под объект (ссылка ещё указывает на «пустую» память)
2. Инициализировать объект — выполнить конструктор
3. Записать ссылку в `instance`

JIT и CPU вправе переупорядочить шаги 2 и 3: сначала записать ссылку (шаг 3), потом выполнить конструктор (шаг 2). Внутри потока, создающего объект, это незаметно — он использует объект только после `new`. Но другой поток в строке (1) видит `instance != null` (ссылка уже записана) и возвращает частично инициализированный объект — конструктор ещё не завершился.

Это не теоретическая угроза. На x86 этот reorder практически не происходит из-за TSO, на ARM — реален. В серверных приложениях с кросс-платформенным деплоем это работало на dev-машине и падало на продакшн-сервере с другой архитектурой.

**Исправление**: объявить `instance` как `volatile`. Volatile-запись в поле гарантирует HB со всеми предшествующими операциями, включая шаги 1 и 2. JIT больше не вправе вынести запись ссылки до конструктора.

```java
// Правильно:
private static volatile Singleton instance;
```

Альтернатива — initialization-on-demand holder idiom, которую многие считают чище:

```java
class Singleton {
    private Singleton() {}

    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

Здесь безопасность обеспечивает class initialization lock: JVM гарантирует, что статический инициализатор выполнится ровно один раз под встроенным локом до того, как класс станет доступен. Никакого explicit `volatile` или `synchronized` не нужно.

## 5. Final поля и safe publication

`final` — это не просто константа. У него специальная семантика в JMM.

**Гарантия JMM**: после завершения конструктора все потоки гарантированно видят корректно инициализированные `final` поля — без какой-либо синхронизации. JVM реализует это через freeze action: при выходе из конструктора вставляется StoreStore барьер для final полей.

```java
class ImmutablePoint {
    final int x, y;
    ImmutablePoint(int x, int y) { this.x = x; this.y = y; }
}
```

Если другой поток получает ссылку на `ImmutablePoint` после завершения конструктора — через любой механизм — он гарантированно читает правильные `x` и `y`. Именно на этом основана безопасность неизменяемых объектов (`String`, записи `record`, коллекции из `List.of()`).

**Важное ограничение**: если ссылка на объект утекла из конструктора (this escape), гарантия нарушается. Типичная ловушка — регистрация `this` в статическом реестре внутри конструктора:

```java
class Broken {
    final int value;
    Broken(int v) {
        Registry.register(this); // утечка this до инициализации value
        this.value = v;
    }
}
```

**Safe publication** — паттерн безопасной передачи ссылки на объект между потоками. «Безопасная» публикация означает, что другой поток не увидит частично инициализированный объект.

Способы безопасной публикации:
- Через `volatile` поле или `AtomicReference`
- Через `synchronized` блок
- Через `static` инициализатор (class loading guarantee)
- Через `final` поле другого уже безопасно опубликованного объекта

**Unsafe publication**: создать объект и записать ссылку в обычное нефинальное нестатическое поле без синхронизации. Другой поток может увидеть ненулевую ссылку на объект с дефолтными значениями полей (0 / null), потому что запись ссылки переупорядочилась раньше записей полей.

## 6. False Sharing и кэш-линии

Это тема, которая не про JMM напрямую, но неизбежно возникает в разговоре о многопоточной производительности.

CPU кэш работает строками (cache line) — типично 64 байта. Если два потока работают с разными переменными, но они лежат в одной cache line, возникает false sharing (ложное совместное использование): запись одной переменной инвалидирует cache line у другого CPU, хотя другой поток вообще не трогает эту переменную. CPU тратит время на cache coherence protocol (MESI/MESIF: Modified, Exclusive, Shared, Invalid), хотя логического конфликта нет.

```java
// Потенциальный false sharing:
long[] counters = new long[4];
// Thread 1: counters[0]++
// Thread 2: counters[1]++
```

`counters[0]` и `counters[1]` — 8 байт каждый, вместе 16 байт, оба в одной 64-байтной cache line. Каждая запись T1 инвалидирует cache line у T2 и наоборот. В микробенчмарках это даёт деградацию в 5–10x по сравнению с раздельными cache line.

**`@Contended`** (аннотация `jdk.internal.vm.annotation.Contended`, доступная через `--add-opens` или в JDK-коде) добавляет padding вокруг поля/класса, выравнивая его на отдельную cache line. Именно так устроены `AtomicLong`, `LongAdder` (внутренний `Cell`), счётчики в `ForkJoinPool`, поле `threadLocalRandomSeed` в `Thread`.

Для пользовательского кода без `@Contended` — ручной padding:

```java
// Грубо, но работает:
class PaddedCounter {
    long p0, p1, p2, p3, p4, p5, p6; // 56 байт padding
    volatile long value;               // 8 байт — отдельная cache line
    long p8, p9, pa, pb, pc, pd, pe;  // ещё 56 байт
}
```

На практике false sharing диагностируется через async-profiler с режимом `perf` — видны аномально высокие LLC-load-misses на конкретных адресах.

## 7. Synchronized vs Volatile vs AtomicXxx: что выбрать

| Механизм | Видимость | Атомарность | Упорядочивание | Накладные расходы |
|---|---|---|---|---|
| `volatile` | да | только одиночное чтение/запись | запрет переупорядочивания через поле | минимальный |
| `synchronized` | да | монопольный доступ к блоку | полный | умеренный (biased/thin/fat lock) |
| `AtomicInteger` / `AtomicReference` | да | CAS-операции | да | минимальный при low contention |
| `LongAdder` | да | да | да | лучше AtomicLong при high contention |

`synchronized` при выходе из блока не просто отпускает лок — он flush'ит все изменения. При входе в следующий synchronized на том же мониторе другой поток видит всё, что было сделано внутри предыдущего блока. Это HB правило #2.

`LongAdder` — оптимизация для high-contention счётчиков: каждый поток пишет в свою ячейку (разные cache line), `sum()` собирает итог. Нет единой точки конкуренции. Используй вместо `AtomicLong` когда счётчик обновляется очень часто из многих потоков, а точное значение между инкрементами не нужно.

## Каверзные вопросы на интервью

**Чем `volatile` отличается от `synchronized`?**

`volatile` даёт видимость одного поля и запрет переупорядочивания через него. Никакого взаимного исключения — два потока могут одновременно проходить через volatile-запись. `synchronized` даёт взаимное исключение (только один поток в блоке) плюс видимость всего, что изменено внутри блока. Если нужна атомарность составной операции — `synchronized` или `AtomicXxx`.

**`volatile long` атомарен?**

Да, по спецификации JLS. Без `volatile` запись `long` (64 бита) может быть двумя 32-битными операциями на 32-битных платформах — word tearing. `volatile long` гарантирует атомарность чтения и записи. На 64-битных JVM это справедливо практически и без `volatile`, но спецификация этого не гарантирует.

**Что гарантирует static final инициализатор?**

Class initialization в JVM защищена встроенным локом: статический инициализатор выполняется ровно один раз до первого использования класса любым потоком. На этом основан Holder idiom для lazy singleton — безопасно без explicit `volatile`.

**Как JMM взаимодействует с моделью памяти CPU?**

x86 использует TSO (Total Store Order): единственный легально возможный reorder это StoreLoad (запись может задержаться, пока её не «увидели» последующие нагрузки). ARM использует слабую модель памяти: все четыре типа переупорядочивания (LoadLoad, LoadStore, StoreLoad, StoreStore) возможны. JMM абстрагирует это — правила одни для всех платформ. JIT на ARM вставляет больше барьерных инструкций для соблюдения JMM-гарантий. Отсюда: баги JMM, которые «никогда не воспроизводились», начинают воспроизводиться на ARM серверах (AWS Graviton, Ampere Altra).

**Что такое roach motel reordering?**

Компилятор по JMM вправе перемещать операции внутрь критической секции (в «мотель»), но не вправе выносить их наружу (из «мотеля» не выбраться). Говоря точнее: операции до `lock` могут быть перемещены после `lock`, но операции после `unlock` не могут быть перемещены до `unlock`. Это легальный реордер — важно знать, что монитор не защищает от переупорядочивания снаружи.

**Почему double-checked locking без `volatile` нельзя исправить добавлением дополнительных локов?**

Потому что проблема в том, что первая проверка `instance == null` выполняется без лока намеренно — для производительности. Никакой лок не защитит чтение, которое сознательно вынесено за его пределы. Правильное решение — сделать само поле `instance` volatile, чтобы его чтение и запись имели нужные барьеры независимо от локов.
