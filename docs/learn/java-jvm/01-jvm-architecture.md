# Архитектура JVM: class loading, runtime data areas, execution engine

JVM — не просто «среда исполнения Java». Это виртуальная машина со своей спецификацией,
несколькими независимыми реализациями и чётко разделёнными подсистемами. Понимание её
устройства объясняет поведение программы в ситуациях, где интуиция молчит: почему первые
запросы медленнее, почему один класс может быть загружен дважды, и откуда берётся
OutOfMemoryError на Metaspace при казалось бы нормальном heap.

## JVM: спецификация vs реализация

JVM определяется **спецификацией** (JVMS — Java Virtual Machine Specification). Спецификация
описывает формат `.class`-файла, набор инструкций байткода, семантику исполнения, правила
верификации. Всё что в ней не закреплено — остаётся на усмотрение реализации.

Реализации:
- **HotSpot** (Oracle JDK, OpenJDK) — доминирующая реализация. Всё что написано ниже
  относится к HotSpot если не указано иное.
- **OpenJ9** (Eclipse, IBM) — альтернативная реализация с другими trade-off'ами по памяти
  и startup (активно применяется в enterprise-окружениях IBM).
- **GraalVM CE** — включает HotSpot JIT + Graal Compiler как замену C2, а также
  возможность AOT-компиляции в native-image.

Важный момент: JVM не привязана к Java. Любой язык, компилятор которого генерирует
валидный байткод, исполняется на JVM. Kotlin, Scala, Clojure, Groovy — все они работают
поверх той же спецификации.

## Class Loading Subsystem

Путь класса от `.class`-файла до готового к исполнению состояния проходит три фазы.

### Loading (загрузка)

ClassLoader находит байты класса (из classpath, modulepath, сети, базы данных — откуда
угодно), читает их и создаёт объект `java.lang.Class` в **method area** (Metaspace в
HotSpot). На этом этапе суперклассы и интерфейсы тоже загружаются рекурсивно.

### Linking (связывание)

Три под-этапа:

**Verification (верификация)**: JVM проверяет что байткод корректен — типы совместимы,
переходы ведут в допустимые точки, стек не переполнится. Это важная линия защиты: ClassLoader
может подсунуть произвольный байткод, верификатор отсекает невалидный. Верификация —
дорогостоящая операция, поэтому результат кэшируется.

**Preparation (подготовка)**: статические поля получают **дефолтные значения** — `0`, `false`,
`null`. Не значения из инициализаторов! Инициализаторы выполняются позже.

**Resolution (разрешение символьных ссылок)**: символьные ссылки константного пула
(имена классов, методов, полей в виде строк) заменяются прямыми ссылками (указателями).
Может быть lazy — откладывается до первого использования.

### Initialization (инициализация)

Выполняется `<clinit>` — синтетический метод, собранный из всех `static {}` блоков и
статических инициализаторов в порядке исходного кода. JVM гарантирует что `<clinit>`
выполняется **ровно один раз** и **потокобезопасно** — благодаря внутренним блокировкам.
Это основа безопасного Singleton через holder-class.

Класс инициализируется при **первом активном использовании**: создание экземпляра,
вызов статического метода, доступ к статическому полю (кроме константных `static final`
примитивов — их компилятор подставляет inline). Загрузка класса ≠ инициализация.

### Две распространённые ошибки, которые путают

**`ClassNotFoundException`** — бросается при явной попытке загрузить класс (`Class.forName()`,
`ClassLoader.loadClass()`), когда ClassLoader не нашёл байты. Checked exception.

**`NoClassDefFoundError`** — бросается когда класс был виден на этапе компиляции, но
недоступен в runtime. Самая частая причина — dependency hell: скомпилировали с версией
`foo-1.0.jar`, а в runtime оказалась `foo-2.0.jar` без нужного класса. Error, не Exception.

Ещё один нюанс: если `<clinit>` бросил исключение, класс помечается как «errored» и
все последующие попытки использовать его бросают `ExceptionInInitializerError`, а потом
`NoClassDefFoundError` — даже если первоначальное исключение уже нигде не отображается.

## Runtime Data Areas

Память JVM разделена на несколько областей с разными правилами жизни.

```
┌─────────────────────────────────────────────────────┐
│                    JVM Process                       │
│                                                     │
│  ┌──────────────┐  ┌──────────────────────────────┐ │
│  │   Heap       │  │  Non-Heap / Native Memory    │ │
│  │  (Young+Old) │  │  ┌──────────┐ ┌───────────┐ │ │
│  │              │  │  │Metaspace │ │Code Cache │ │ │
│  └──────────────┘  │  └──────────┘ └───────────┘ │ │
│                    └──────────────────────────────┘ │
│  Per-thread:                                        │
│  ┌──────────┐ ┌──────────┐ ┌───────────────────┐  │
│  │JVM Stack │ │PC Reg    │ │Native Method Stack│  │
│  └──────────┘ └──────────┘ └───────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### Heap

Разделяется всеми потоками. Здесь живут экземпляры объектов и массивы. Размер управляется
`-Xms` (начальный) и `-Xmx` (максимальный). Управляется GC (подробно в главе 03).

### Method Area / Metaspace

Структуры загруженных классов: constant pool runtime, описания методов и полей, байткод.
До Java 8 называлась **PermGen** (Permanent Generation) и была частью heap с фиксированным
размером. Overflow → `OutOfMemoryError: PermGen space`.

С Java 8 — **Metaspace**: часть **native memory** (не heap Java). Растёт динамически.
Ограничивается флагом `-XX:MaxMetaspaceSize`; без него — растёт пока не кончится native
память или не вмешается OS killer.

Причина отказа от PermGen: фиксированный размер создавал OOM при интенсивной динамической
генерации классов. Hibernate, Spring, CGLIB, JSP-компиляция — все они генерируют классы
на лету. PermGen постоянно нужно было тюнить под приложение; Metaspace снял эту проблему.

### Code Cache

JIT-скомпилированный машинный код хранится в Code Cache — отдельной области native memory.
Ограничивается `-XX:ReservedCodeCacheSize` (default ~240MB в Java 11+). Если Code Cache
переполнится — JVM переключается обратно на интерпретацию и выводит предупреждение. В
production это незаметный, но болезненный деградирующий сценарий.

### JVM Stack

Каждый поток имеет свой JVM Stack. Содержит **кадры (frames)**: по одному на каждый
вызов метода. Кадр содержит: массив локальных переменных, стек операндов, ссылку на
constant pool текущего класса. При рекурсии без базы — `StackOverflowError`.

Размер стека потока: `-Xss` (default 512KB–1MB в зависимости от платформы). Важно при
создании тысяч потоков — каждый резервирует `-Xss` native memory.

### PC Register и Native Method Stack

**PC Register** (Program Counter) — per-thread, указывает на текущую bytecode-инструкцию.
Для native методов — undefined (нативный код не в JVM стеке).

**Native Method Stack** — аналог JVM Stack для JNI-методов, на native стороне.

## Execution Engine

### Интерпретатор

Читает bytecode инструкцию за инструкцией и выполняет их. Запускается мгновенно, но
медленно. Одновременно собирает **профиль**: сколько раз вызван метод, какие типы
встречались в instanceof/checkcast, какие ветки чаще берутся.

### JIT Compiler

Когда метод становится достаточно «горячим» — JIT компилирует его в машинный код. HotSpot
содержит два JIT-компилятора:

- **C1** (Client Compiler): быстрая компиляция, умеренные оптимизации. Используется на
  ранних уровнях прогрева.
- **C2** (Server Compiler): агрессивные оптимизации на основе накопленного профиля.
  Требует времени на компиляцию, даёт максимальный gain.

Детально — в главе 04.

## Lifecycle: от `java` до первого запроса

1. OS запускает процесс `java`
2. JVM инициализирует bootstrap ClassLoader, загружает `rt.jar` / модули платформы
3. Загружается главный класс (Application ClassLoader)
4. Выполняются все `<clinit>` на пути к `main()`
5. `main()` начинает работу — JVM в режиме интерпретатора
6. Горячие методы постепенно компилируются C1 → C2

Первые запросы медленнее не абстрактно — конкретно: горячие пути ещё не скомпилированы C2,
профиль не накоплен. В нагрузочном тестировании это обязательно учитывать: бенчмарк без
warmup периода измеряет не production-производительность, а производительность прогрева.

## Каверзные вопросы интервью

**Когда выполняется `static` блок?** При инициализации класса, не при загрузке. Класс
можно загрузить (`Class.forName("Foo", false, cl)`) без инициализации. Инициализация
происходит при первом активном использовании.

**Может ли один класс быть загружен дважды?** Да — разными ClassLoader'ами. Два объекта
`Class` для одного имени, в разных namespace. Это основа изоляции плагинов (OSGi),
hot-reload в app-серверах, и источник `ClassCastException` вида «не могу привести Foo к Foo»
когда оба класса загружены разными лоадерами.

**Чем Metaspace отличается от heap?** Metaspace — native memory, не учитывается в `-Xmx`,
не управляется Java GC напрямую (освобождается при unload классов). OOM на Metaspace —
отдельный от heap, требует отдельного мониторинга.

**Почему убрали PermGen в Java 8?** Фиксированный размер PermGen создавал регулярные OOM
при интенсивном использовании динамической генерации классов. Каждое приложение требовало
тюнинга `-XX:MaxPermSize`. Metaspace в native memory убрал это ограничение — классы
выгружаются при unload ClassLoader'а, native память возвращается ОС.
