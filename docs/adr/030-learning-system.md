# ADR-30: Learning system — материалы и тесты как dual-mode артефакт

- Статус: Принято
- Дата: 2026-04-25
- Авторы: Sergey Knyazev, Claude Opus 4.7
- Теги: learn, content, skills, rag, pdf, architecture

## Контекст

Готовлюсь к интервью на Tech Lead / Architect / Staff+. Нужны:
- учебные материалы для самоподготовки по ключевым областям (System Design, Kotlin, архитектура, leadership, DS&A);
- тесты для самопроверки (multiple choice, fill-in, freeform с LLM-валидацией);
- адаптивность: при провале пробел закрывается дополнительным материалом;
- возможность сослать интервьюера на этот же материал;
- компиляция в PDF-книгу как самодостаточный артефакт.

Существующие части проекта (`docs/skills/` для CV, `docs/projects/`, `docs/experience/`) описывают *уже накопленный* опыт. Здесь нужен другой жанр: *активное обучение* + верификация знаний. Это не extension к ADR-026 (skills как роутинг RAG), а отдельный слой с собственной семантикой.

Главный конфликт интересов:
- **Для меня** система должна быть мутабельной — провалы записываются, прогресс трекается, материалы дописываются под мои пробелы.
- **Для интервьюера** — иммутабельной: проходит тесты, видит результаты, ничего не меняется в репо.

Если строить «один UX для всех», получается либо разрешить запись с веба (auth + риски), либо лишить себя адаптивности. Решение — структурное разделение по среде запуска, а не по правам доступа.

## Решение

### Dual-mode архитектура

Один источник данных (`docs/learn/`), два потребителя:

| Среда | Кто | Доступ к данным | Запись |
|-------|-----|-----------------|--------|
| **Skill-режим** (Claude Code, локально) | автор | full | да: `remediation/generated/`, `.progress.json` |
| **Web-режим** (браузер на /learn) | интервьюер | read-only через бэкенд | нет |

Иммутабельность для гостя обеспечена *структурно*: у веба нет write-эндпоинтов на `docs/learn/`. Не нужна авторизация — это свойство архитектуры, а не политика.

### Темы независимы — нет порядка прохождения

Каждая тема в `docs/learn/<topic>/` самодостаточна. Глобального маршрута «сначала это, потом то» нет. `docs/learn/README.md` — **каталог**, не учебный план. Skill `study` показывает темы как «выбирай что хочешь», а не «следующее».

Это сознательное решение: цель — *наличие* материалов по каждой теме, не *прогрессивное обучение*. У темы внутри может быть свой порядок глав (`01-`, `02-`, ...) — это локальное удобство, не глобальный контракт.

### Структура данных

```
docs/learn/
  README.md                              — каталог тем
  <topic>/                               — например, system-design/
    README.md                            — карта темы
    01-fundamentals.md                   — главы (свободный порядок имён)
    02-scalability.md
    ...
    tests/
      01-mcq.md                          — multiple choice
      02-fill.md                         — fill-in
      03-freeform.md                     — freeform + rubric
    remediation/
      static/<qid>.md                    — курированные пояснения
      generated/<YYYY-MM-DD>-<qid>.md    — LLM-пояснения для автора
  .progress.json                         — gitignored, прогресс автора
```

### Формат теста

Frontmatter определяет тип и валидацию:

```markdown
---
test_id: sd-01-mcq
topic: system-design
type: multiple_choice
questions:
  - id: q1
    text: "Какая консистентность у DynamoDB по умолчанию?"
    options: [strong, eventual, causal, snapshot]
    answer: eventual
    remediation: sd-q1
---
# Пояснение к тесту
Опциональный обзорный текст. На прохождение не влияет.
```

Типы тестов:

| `type` | Валидация | Стоимость |
|--------|-----------|-----------|
| `multiple_choice` | сравнение строк | 0 |
| `fill_in` | regex или exact match (поле `pattern`) | 0 |
| `freeform` | LLM с rubric (поле `rubric`), низкая температура | 1 LLM-вызов на ответ |

Поле `remediation` указывает id; пояснение ищется в `remediation/static/<id>.md` (если есть) или генерируется LLM в `remediation/generated/<date>-<qid>.md` (только в skill-режиме).

### Адаптивность: LLM генерирует, потом курируется вручную

При провале вопроса в skill-режиме:

1. Skill спрашивает у автора **обоснование** ответа (для freeform — оно уже дано; для MCQ/fill — задаётся отдельно: «почему ты выбрал X?»).
2. LLM получает контекст: вопрос + ответ + обоснование + эталон + текст соответствующих глав темы.
3. Генерируется адресное пояснение — **именно то, что автор упустил**, а не общее изложение темы.
4. Записывается в `remediation/generated/<YYYY-MM-DD>-<qid>.md` с frontmatter (`qid`, `date`, `attempt`).
5. Прогресс обновляется в `.progress.json`.

`generated/` сохраняется навсегда — это learning log автора, не временные файлы. Файлы коммитятся в репо.

**Промоушен `generated/ → static/`** — отдельный skill `promote`, ручной. Автор просматривает свежий `generated/<file>`, решает, достоин ли материал войти в постоянный курикулум. Если да — skill переносит в `static/`, опционально интегрирует в основные главы. Автоматизации нет осознанно: качество промоутится человеком.

### PDF-сборка — один артефакт

`learn-book.pdf` собирается командой `bun run build:learn`. Содержит:

1. Обложка + введение
2. Темы (произвольный порядок, например, алфавитный) — главы + статические remediation как часть темы
3. Приложение «Дневник пробелов» — `generated/`, отделённое визуально от курикулума

Один PDF, потому что `generated/` уже коммитится и публично — отдельный «приватный» PDF был бы фикцией. Включение `generated/` усиливает артефакт: показывает не только материал, но и процесс обучения автора.

### Skills

Три skill'а в `.claude/skills/`:

| Skill | Назначение |
|-------|------------|
| `quiz` | Принимает `<topic>` или `<test-file>`, проводит тест, при провале вызывает LLM-remediation, обновляет прогресс |
| `study` | Каталог тем, статус прохождения, рекомендация следующего теста *в рамках темы* (без глобального маршрута) |
| `promote` | Просмотр `generated/`, перенос в `static/` или в основные главы |

Skills предназначены для автора (Claude Code). Веб использует другие механизмы — Ktor-эндпоинт.

### Web-эндпоинт

`POST /api/learn/validate` — принимает `{topic, test_id, qid, answer}`, возвращает `{correct: bool, explanation: string?}`. Для freeform использует rubric из теста и LLM-валидатор (один вызов с кэшируемым префиксом — system + rubric).

Никаких write-эндпоинтов. Прогресс гостя — в `localStorage` браузера, не на сервере.

### Интеграция с RAG

`docs/learn/` автоматически попадает в индекс через `DocumentLoader` (ADR-016). Главы становятся частью корпуса: агент на главной может отвечать со ссылками на учебные материалы. Никакого дополнительного кода — только убедиться, что чанкер не давится новой структурой (тесты в frontmatter, директория `remediation/`).

## Обоснование

- **Структурная иммутабельность вместо authz.** Гарантия «гость ничего не меняет» обеспечена отсутствием write-эндпоинтов на стороне веба. Это не политика, которую можно случайно ослабить, а свойство кода.
- **Один источник, два рендера.** Тесты, главы и remediation существуют как md-файлы. Skill и веб — два разных читателя одних и тех же данных. Дублирования нет.
- **Темы независимы — соответствует цели.** Цель проекта — *наличие* материалов по ключевым областям, а не *учебный курс*. Жёсткий порядок прохождения создавал бы фиктивную педагогику и ограничивал гостя.
- **LLM-генерация remediation, а не подбор существующих.** Подбор материалов на основе ошибки — это retrieval-задача с теми же ограничениями (ADR-019). Генерация адресного пояснения по контексту ошибки даёт лучший сигнал. Цена: один LLM-вызов на провал, что приемлемо в skill-режиме.
- **`generated/` как артефакт, не как мусор.** Treatment generated-материалов как «временных файлов» обесценил бы их. Сохранение целиком + коммит в репо превращает дневник пробелов в публичное свидетельство процесса обучения — для staff+ это сильный сигнал.
- **Ручной промоушен, не автоматический.** Автоматический promote по N попаданий привёл бы к деградации курикулума: low-quality LLM-пояснения попадали бы в постоянный материал. Ручной шаг сохраняет качество `static/`.
- **PDF без приватной сборки.** Если `generated/` коммитится — он публичный. Делать «приватный» PDF — обманывать себя об уровне приватности. Один артефакт, честный.
- **Композиция со skills/RAG, не замена.** ADR-026 описывает skills как роутинг для агента (server-side). ADR-030 описывает skills для автора (client-side, Claude Code). Оба слоя независимы, путаются только через имя «skill» — путаница допустима, оба контекста сохраняем.

## Последствия

### Положительные
- Учебные материалы публично верифицируемы, обновляются естественным процессом обучения.
- Артефакт усиливается со временем: каждое моё провальное прохождение → коммит в `generated/` → демонстрация процесса.
- Интервьюер может пройти тесты сам, без локального запуска.
- `docs/learn/` автоматически попадает в RAG — агент использует учебные главы при ответах посетителям.
- PDF-книга — самодостаточный оффлайн-артефакт, его можно отправить вместе с CV.
- Темы развиваются независимо: одну можно прорабатывать глубоко, другая остаётся на уровне базовых глав без блокировки общей системы.

### Отрицательные / Компромиссы
- Дублирование UX: skill-интерфейс (терминал, диалог) и веб-интерфейс (форма) — два разных опыта прохождения теста. Гарантировать, что веб даст «такой же» опыт, как skill — невозможно. Решение: явно зафиксировать, что skill-режим — авторский (с обоснованиями, обучением), веб — гостевой (формальная проверка).
- LLM-валидатор freeform-вопросов на вебе платится при каждом прохождении. Кэширование префикса (system + rubric) снижает стоимость, но не до нуля.
- `generated/` со временем разрастётся. Это сознательная плата за полноту артефакта; реорганизация (архив старых) делается отдельным шагом, если понадобится.
- Валидация freeform-ответов rubric'ом субъективна. Два LLM-вызова на одинаковый ответ могут дать разные вердикты. Смягчение: rubric пишется детально, температура валидатора 0.1.
- Темы без жёсткого порядка → нет «учебной траектории» для гостя. Это компромисс к цели: артефакт — каталог знаний, не курс.
- `generated/` зависит от того, что автор будет регулярно проходить тесты. Если автор перестанет — артефакт остановится в развитии. Это структурное свойство, не баг.
- Удвоение слова «skill» в проекте: server-side skills (ADR-026) и client-side skills (ADR-030). Снимается контекстом и явным указанием в документации.

## Рассмотренные альтернативы

### Только skill, без веба
Отклонено: интервьюер не сможет проверить заявления автора без клонирования репо. Это снижает ценность тестов как артефакта.

### Только веб, без skills
Отклонено: пропадает мутабельность для автора. Без записи `generated/` адаптивность работает только в рамках одной сессии и не накапливается. Артефакт «дневник пробелов» становится невозможен.

### Запись с веба + auth для автора
Отклонено: добавляет auth-инфраструктуру, риск мисконфига (гость случайно с автор-правами) и противоречит структурной иммутабельности. Skill-путь даёт мутабельность бесплатно через локальное окружение.

### Жёсткий порядок прохождения тем
Отклонено по уточнению автора. Цель — каталог знаний, не курс. Жёсткий порядок ограничивал бы гостя, который, например, хочет проверить только «System Design», и создавал бы ложную педагогику.

### LLM-генерация remediation на лету, без сохранения
Отклонено: пояснение, существующее одну сессию, не становится артефактом. Сохранение в `generated/` превращает каждый провал в коммит — публичное свидетельство процесса.

### Автоматический промоушен `generated/ → static/`
Отклонено: качество `static/` деградирует. LLM-пояснение хорошо для one-off обучения, но как канонический материал требует курирования.

### Два PDF (книга + личный лог)
Отклонено: дублирование. Если `generated/` коммитится, он публичный — приватный PDF был бы фикцией. Один PDF, всё включено, визуально разделено на курикулум и приложение.

### `docs/learn/` отдельным репо или субмодулем
Отклонено: теряется автоматическая интеграция с RAG (ADR-016) — пришлось бы дописывать загрузчик. Учебные материалы — часть общего корпуса проекта, держать их в одном репо естественно.

### Тесты как JSON, не markdown с frontmatter
Отклонено: markdown с frontmatter единообразен с остальным контентом проекта (ADR-013), редактируется в любом редакторе, читается человеком без парсера. JSON выиграл бы только при сложной валидации, которой здесь нет.

### Подбор существующих материалов вместо генерации remediation
Отклонено: подбор — retrieval-задача со всеми её ограничениями. Генерация даёт **адресное** пояснение того, что автор упустил, а не общее повторение темы.

## Связанные артефакты

- `docs/learn/` — **новая директория**, контент учебных материалов и тестов
- `docs/learn/README.md` — **новый**, каталог тем
- `docs/learn/<topic>/...` — **новые**, главы, тесты, remediation по темам
- `.claude/skills/quiz/` — **новый**, skill проведения тестов
- `.claude/skills/study/` — **новый**, skill навигации по темам
- `.claude/skills/promote/` — **новый**, skill промоушена `generated/ → static/`
- `scripts/build-learn.ts` — **новый**, сборка `learn-book.pdf` (по аналогии с `build-cv.ts`, см. ADR-014)
- `app/public/learn-book.pdf` — **gitignored**, выходной артефакт сборки
- `server/src/commonMain/kotlin/dev/knyazev/api/LearnRoutes.kt` — **новый**, эндпоинт `POST /api/learn/validate`

## Связь с другими ADR

- **ADR-002** (source of truth) — учебные материалы как первичный markdown без промежуточных представлений; тесты, главы, remediation существуют единственный раз, два потребителя их читают.
- **ADR-013** (content structure) — `docs/learn/` следует общим конвенциям markdown + frontmatter, попадает в общий контентный слой.
- **ADR-014** (CV PDF generation) — компилятор `learn-book.pdf` строится по аналогии с `build-cv`, переиспользует тулинг.
- **ADR-016** (RAG pipeline) — `docs/learn/` автоматически индексируется `DocumentLoader`, агент получает доступ к учебным главам без дополнительного кода.
- **ADR-026** (Skills layer) — server-side skills для роутинга агента; ADR-030 описывает client-side skills (Claude Code) для автора. Разные слои, общее имя.
- **ADR-027** (Agent modes) — учебные материалы доступны во всех режимах (agentic/full-context/rag), потому что попадают в общий корпус.
