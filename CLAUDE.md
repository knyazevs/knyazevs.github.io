# CLAUDE.md — Правила работы с этим репозиторием

Этот файл описывает обязательные соглашения для всех, кто работает с репозиторием —
включая AI-ассистентов. Соглашения отражают цель проекта: демонстрация мышления и
рабочего процесса автора будущему работодателю.

---

## Цель проекта

Трудоустройство на роль Technical Lead / Architect / Staff+.
Проект — публично верифицируемый артефакт, демонстрирующий:
- архитектурное мышление через ADR
- инженерные решения через код
- рабочий процесс через историю коммитов и блог
- личный нарратив через профиль и блог

---

## Архитектура проекта (три слоя)

```
docs/        — контентный слой: ADR, блог, профиль, опыт, навыки
app/         — визуальный слой: Astro + Svelte 5, WebGL-орб, чат-интерфейс
server/      — интеллектуальный слой: Kotlin/Ktor, RAG-пайплайн (HyDE + BM25 + RRF)
scripts/     — инфраструктурные скрипты (генератор CV)
```

### Страницы

| URL | Описание |
|-----|----------|
| `/` | Главная: орб по центру, чат с агентом, подсказки |
| `/code` | Браузер кода: дерево репозитория + просмотр файлов |
| `/timeline` | История: git log с типами коммитов |

### RAG-пайплайн

Агент индексирует весь репозиторий двумя загрузчиками:
- `DocumentLoader` — читает `docs/` рекурсивно, чанкует по H2/H3
- `CodeLoader` — читает `server/src/`, `app/src/`, `scripts/`, конфиги; чанкует по объявлениям функций/классов

Путь кодового чанка в breadcrumb начинается с `code/` (например, `code/server/src/.../RagPipeline.kt`).
Путь документального чанка — относительно `docs/` (например, `experience/001-low-code-platform.md`).

После ответа агент эмитирует SSE-событие `{"sources":[...]}` со списком источников.
Фронтенд показывает чипы под ответом — клик открывает DocumentModal или браузер кода.

### CV-генератор

```sh
bun run build:cv   # → app/public/cv.pdf
```

Читает данные из `docs/experience/`, `docs/skills/`, `docs/profile/`.
Поля с `cv_skip: true` в frontmatter пропускаются.
Выходной файл игнорируется git (`.gitignore`).

## Учебные материалы (docs/learn/)

Параллельно с CV-материалами проект содержит учебные главы и тесты для подготовки
автора к интервью. Это не учебный курс — каталог независимых тем. Полное описание
архитектуры в ADR-030.

### Dual mode

Один источник (`docs/learn/`), два потребителя:

- **Skill-режим** (Claude Code, локально): автор; пишет в
  `<topic>/remediation/generated/` при провале вопросов теста
- **Web-режим** (браузер `/learn`): интервьюер; read-only через бэкенд; никаких
  write-эндпоинтов на `docs/learn/`

Иммутабельность для гостя обеспечена структурно — отсутствием write-API на
стороне веба, не authz.

### Структура темы

```
docs/learn/<topic>/
  README.md                              — карта темы
  NN-короткое-название.md                — главы (свободный порядок имён внутри темы)
  tests/
    NN-...-mcq.md                        — multiple choice (frontmatter с questions)
    NN-...-freeform.md                   — freeform с rubric для LLM-валидатора
  remediation/
    static/<qid>.md                      — курированные пояснения
    generated/<YYYY-MM-DD>-<qid>.md      — LLM-пояснения по провалам автора (коммитятся в репо)
```

Темы независимы — нет глобального порядка прохождения. Внутри темы порядок глав
свободный.

### Формат главы

Глава 1500-2500 слов, 3-4 секции. Каждая секция следует схеме:

1. **Теория** — концепт + формула с интуицией (не падает с неба, выводится через
   смысловое разложение, например `ρ/(1-ρ)` объясняется через «запас (1-ρ) на
   разгребание очереди»)
2. **На практике** — типичный сценарий, разбор «решений в лоб» (явно сказано —
   работает или нет и почему), конкретные рекомендации
3. **Когда модель усложняется** (опционально) — multi-worker, тяжёлые хвосты,
   бёрсты, реальные осложнения базовой модели
4. **Где деньги** (опционально) — экономические соображения, портфельные подходы

### Стилистические правила

Цель формата — средний путь между академическим depth-bombing и story-driven
упрощением:

- **Жаргон определяется на месте** при первом использовании. M/M/1, Poisson,
  Lamport, MVCC, Paxos — любой термин расшифровывается в той же секции, не
  отсылает в смежные темы
- **Формула выводится из интуиции** — «вот откуда берётся (1-ρ) в знаменателе»
  вместо «вот формула, поверь»
- **«Решения в лоб» разбираются явно** — если читатель подумал «настроить
  autoscaling агрессивнее», глава говорит «частично верно, но не главное» и
  объясняет почему недостаточно и что реально работает
- **Кейсы — только если уверен.** Discord Cassandra→ScyllaDB, AWS Spanner
  TrueTime — да. Выдуманные «Slack 2017» — нет; generic-сценарий («типичная
  история: сервис в e-commerce…») лучше выдумки
- **Не name-drop** — упоминать авторов/годы только если действительно полезно
  (Marc Brooker про queueing — да; «Lamport 1978» в reading-list-декорации — нет)
- **ASCII-графики** там, где помогают понять форму (асимптоты, кривые с
  переходами). При PDF-сборке заменятся на рендер из табличных данных

### Когда писать новую главу

При проработке новой темы либо при углублении существующей. Каждая глава —
отдельный коммит `feat(learn): <тема> — <короткое описание>`.

При провале теста skill `quiz` сам пишет remediation в `generated/` — это не
ручной коммит, а часть сессии прохождения теста.

## Блог — обязательный артефакт

### Когда писать запись в блог

Запись в блог обязательна при:
- любом архитектурном решении (даже если есть ADR)
- изменении концепции или переосмыслении подхода
- отладке нетривиальной проблемы
- любом моменте «я думал одно, оказалось другое»
- завершении значимой фичи или итерации

### Формат записи

Файл: `docs/blog/YYYY-MM-DD-короткое-название.md`

```markdown
# Заголовок (что произошло)

- Дата: YYYY-MM-DD
- Теги: тег1, тег2

## Контекст
Что я делал и почему взялся за это.

## Что думал изначально
Моя исходная гипотеза или подход.

## Что пошло не так / что изменилось
Честно — что не сработало, что удивило, что переосмыслил.

## Решение и вывод
Что в итоге сделал и почему именно так.

## Что бы сделал иначе
Ретроспектива — если бы начинал снова.
```

Тон — живой, от первого лица. Не ADR-стиль. Читатель должен видеть человека, а не документ.

Черновик блог-поста пишет ассистент по контексту сессии (произошедшее, решения, тупики, аргументы). Автор вносит правки: уточняет фактологию, усиливает акценты, выпиливает то, что не звучит своим голосом. Задача ассистента — не имитировать авторский стиль, а дать качественное сырьё: честную структуру событий и рассуждений, которое автору остаётся довести до своего голоса.

## Формат коммитов

Каждый коммит должен быть осмысленным и самодокументирующимся.

### Структура

```
<тип>(<область>): <что сделано — конкретно>

<почему — 1-3 предложения если не очевидно>

Refs: ADR-XX, blog/YYYY-MM-DD-название.md
```

### Типы

| Тип | Когда |
|-----|-------|
| `feat` | новая функциональность |
| `arch` | архитектурное решение, рефактор структуры |
| `fix` | исправление бага |
| `docs` | ADR, блог, профиль, README |
| `infra` | Docker, CI/CD, конфиги |
| `design` | визуальные изменения, CSS, анимации |
| `refactor` | улучшение кода без изменения поведения |

### Примеры хороших коммитов

```
arch(rag): заменить single-stage retrieval на HyDE + BM25 + RRF

Косинусное сходство одного запроса давало нерелевантные чанки из-за
семантического разрыва между вопросом и документом. Трёхслойный пайплайн
устраняет это независимыми сигналами.

Refs: ADR-019, blog/2026-04-14-rag-rethink.md
```

```
feat(agent): добавить source-навигацию через SSE

Агент эмитирует {"sources":[...]} после ответа — фронтенд показывает
кликабельные чипы источников под каждым сообщением.
```

### Плохие коммиты (не допускаются)

```
fix stuff
update
wip
changes
```

## ADR — обязательные правила

- Каждое архитектурное решение фиксируется в `docs/adr/NNN-название.md`
- Нумерация строго последовательная, без пропусков
- Статус: Принято / Отклонено / Устарело
- Обязательный раздел «Альтернативы и причины отказа»
- После написания ADR — написать запись в блог с живым нарративом

---

## Что НЕ делать

- Не делать бессмысленных коммитов типа "fix", "update", "wip"
- Не принимать архитектурные решения без ADR
- Не писать ADR без последующей записи в блог
- Не удалять устаревшие ADR — менять статус на «Устарело»
